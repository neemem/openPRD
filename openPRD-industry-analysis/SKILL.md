# 行业/业务分析

## 概述

**目的**：当需求涉及特定行业时，先分析行业特点，确保方案符合行业规范。

**前置条件**：已完成用户调研，了解目标用户和使用场景。

---

## 触发条件

以下情况需要做行业分析：
- 用户描述的是一个新行业的新需求
- 用户说"[行业] 的管理系统怎么做"
- 用户说"帮我分析 [行业]"
- 系统有明显的行业属性（如医疗、金融、政务、教育等）

---

## 输出格式

> 📖 **完整报告模板** — 9 大章节（行业概况 / 行业特点 / 典型业务流程 / 行业术语表 / 技术要点 / 行业标杆产品 / 对本项目的启示 / 参考资源 / 后续行动）的完整 Markdown 输出模板见 [reference/output-template.md](reference/output-template.md)。

**必须按照上述完整格式输出，不得简化**。任何章节缺失都会导致后续方案设计缺乏行业依据。

---

## 执行流程

1. **识别行业关键词**：从需求中提取行业特征（产品经理提到的"客服"、"医疗"等）
2. **收集行业信息**：通过调研、竞品分析、公开资料等了解行业特点
3. **产出分析报告**：按上述完整格式输出
4. **提炼设计启示**：重点输出"对本项目的启示"章节

## Gotchas (避坑指南)

### G-01: 竞品"有 AI 能力"90% 是接 GPT API,没自研
- **症状**: 写"竞品 X 已支持 AI 智能问答,我们也必须做"——但实际是接了 OpenAI API,提示词都写在 config 里
- **原因**: 竞品官网/PPT 宣传"AI",agent 没去验证技术栈
- **修正**: 必查 3 处验证:(1) 招聘 JD 里有没有"算法工程师/模型训练"岗位(2) 工程博客/技术大会分享(3) GitHub 是否开源推理代码。3 处都没有 = 100% 是 API 包装,可低成本跟进。
- **参考**: examples/01 §经验教训 #4(行业标杆数据需引用验证)

### G-02: 行业市场规模数字单源误差可达 5x,必交叉验证
- **症状**: "中国 CRM 市场规模 2025 年达 200 亿"(引用艾瑞 1 家)——但 IDC/Forrester 数据可能是 80 亿
- **原因**: 不同机构统计口径不同(纯 SaaS vs 含实施服务),单源数字被反复引用
- **修正**: 至少 3 个独立来源(国内 1 + 海外 1 + 协会 1),用区间表述("80-200 亿,口径不同")。单源数字必标"待验证"。
- **参考**: examples/01 §14-行业分析报告 §8 参考资源(必须 15+ 来源)

### G-03: 行业术语表只列"是什么",不列"系统怎么映射"
- **症状**: 报告术语表写"线索(Lead): 潜在客户信息",但没说明"系统在哪个表/哪个字段存"
- **原因**: 业务术语和系统实现两层割裂,开发对照术语表写代码时一脸懵
- **修正**: 每个术语 3 字段:**业务定义 / 系统映射(表名+字段)/ 出现场景**。例:"线索(Lead)" → 表名 `crm_lead`、字段 `{name, phone, source, status}`,出现在 `/leads` 列表 + 详情页。
- **参考**: examples/01 §PRD §6 业务对象 + 12-数据库设计

### G-04: 业务流程图只画"理想路径",不画"异常路径"
- **症状**: 报告画"客户下单→支付→发货→收货"主流程,但漏掉"支付失败/库存不足/部分退款/拒收"4 个异常
- **原因**: 业务人员口述时只说"正常情况",agent 没追问"如果 X 异常呢"
- **修正**: 业务流程图必含 3 类:主流程(实线)/ 异常分支(虚线)/ 兜底流程(兜底态)。每类至少 1 个示例,见 examples/01 §PRD §3 范围(状态机 PENDING→CONTACTED→QUALIFIED→CONVERTED 含 INVALID 异常)。
- **参考**: examples/01 §PRD §3 状态机

**跨子技能常见错误汇总**: 见 reference/16-common-pitfalls.md(由 Sub-agent D 整合,本文件为占位说明)

## 与用户调研的关系

```
用户调研（了解用户）→ 行业分析（了解行业）→ 方案设计（满足用户+行业）
```

两者都是方案设计的前置环节：
- **用户调研**：了解具体用户的需求和约束
- **行业分析**：了解行业通用的特点和最佳实践

两者相互补充，用户调研的输出应作为行业分析的输入。

---

## 完成后

1. **输出完整行业分析报告**（按上述格式，不得简化）到项目目录
2. **向用户确认**分析结果，重点确认"对本项目的启示"章节
3. **可选**：如需了解竞品情况，调用 `Skill("openPRD-competitor-analysis")` 增强竞品分析维度
4. **进入下一阶段**：调用 `Skill("openPRD-document-generation")`

## 5 路并行协调机制（v2 强化）

> **变更说明**（2026-06-03）：从"1 个 subagent"升级为"5 路并行 subagent"，确保行业研究的**深度 + 广度**。

### 5 路 subagent 并行

| 路径 | subagent | 关注维度 | 输出文档 |
|------|----------|----------|----------|
| **A. 行业分析** | `openPRD-industry-analysis`（**本 skill**）| 行业概况、典型流程、术语 | 14-行业分析报告.md |
| **B. 竞品分析** | `openPRD-competitor-analysis` | 竞品功能/定价/定位/SWOT | 14-竞品分析报告.md |
| **C. 标杆研究** | `openPRD-deep-research` | 业界最佳实践、参考架构 | 14-标杆研究.md |
| **D. 法规合规** | `openPRD-regulation-compliance` | 适用法规、合规 Checklist | 14-合规清单.md |
| **E. 技术趋势** | `openPRD-tech-trends` | 3-5 年技术演进、技术选型 | 14-技术趋势.md |

### 协同时序

```
【触发】用户研究 Q1-Q4 完成后
   ↓
【并行启动】5 个 subagent（基于 Q1+Q2 早期需求画像）
   ├─ A. 行业分析  ──┐
   ├─ B. 竞品分析  ──┤
   ├─ C. 标杆研究  ──┤
   ├─ D. 法规合规  ──┤
   └─ E. 技术趋势  ──┘
   ↓
【用户回答 Q5-Q35】继续
   ↓
【回填】5 路 subagent 基于完整需求画像修订
   ↓
【合并产出】5 份独立报告 + 1 份综合洞察文档
```

### 冲突解决

5 路 subagent 的输出可能存在**冲突**：

| 冲突类型 | 解决方案 |
|----------|----------|
| **A vs D（行业惯例 vs 法规）** | 法规优先，行业惯例调整 |
| **B vs D（竞品做法 vs 法规）** | 法规优先，竞品作为参考 |
| **C vs E（当前标杆 vs 未来趋势）** | 当前 + 演进路线并重 |
| **A vs B（行业 vs 竞品差异）** | 行业分析通用，竞品做差异化 |
