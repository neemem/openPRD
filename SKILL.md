## 核心原则

1. **严格遵守流程**:即使用户说"直接开发",也必须先完成调研流程
2. **基于源码验证**:禁止近似值,颜色/字体必须验证源码
3. **Mock数据贴合业务**:数据要像真实业务场景
4. **实用性表现形式**:流程图、表格、重点加粗,提高可阅读性
5. **Multi-Agent并行**:独立任务并行处理(详细见 [reference/10](./reference/10-orchestration-detail.md))
6. **必须等待用户回答**:每次提问后必须等待用户回答才能继续
7. **⚠️ 页面级方案强制要求**:触发词含"页面"时,必须按 16 章 PRD 体系产出
8. **⚠️ 35 题用户调研**:9 组共 35 题,覆盖深度业务/数据合规/运营增长
9. **⚠️ 5 路行业研究并行**:行业 + 竞品 + 标杆 + 合规 + 技术趋势 5 路 subagent 并行
10. **⚠️ 外部能力嵌入**:涉及具体库时调 context7、视觉化时调 minimax、实时网页时调 chrome-devtools
11. **⚠️ 四级质量门控**:G1 形式 / G1.5 产物 / G2 阶段 / G3 全局(详见 [reference/05-quality-gates.md](./reference/05-quality-gates.md);G1.5 产物级验证详见 [reference/17](./reference/17-product-level-verification.md))
12. **⚠️ 产出范围可选化(v4.2)**:Step 0 第四问选 4 类受众(产品/技术/运营/合规),不选的受众对应文档降级为 ≤ 200 字摘要;默认全选,兼容 v4.1 老用户。详见 [reference/19](./reference/19-audience-tag-table.md)
13. **⚠️ Step 0 自适应(v4.4)**:主 Agent 静默自动判断 7 个维度(主题/用户/项目类型/后端前端/页面级/BI/行业),能判的不问,判不准/有歧义才问。每问带"为什么"+"我不确定"兜底。详见下方 Step 0 章节
14. **⚠️ G0 需求清晰度门控(v4.4)**:Step 0 答完后,主 Agent 复述最终画像 + 互斥检测,用户确认后才进 Step 1。详见 [reference/20](./reference/20-g0-requirements-clarity.md)
15. **⚠️ Step 4 用户复盘(v4.4)**:交付完成后用 18-PLR 模板跑复盘,收集"哪不准/哪漏",作为 openPRD 持续改进的数据源

## ⚠️ Step 0 重要前置判断(自适应·v4.4 重构)

**设计原则**:主 Agent 先**静默自动判断**(不打扰),能判的判;判不准/有歧义的才问用户。每问带"为什么"+"我不确定"兜底。

### 自动判断优先级

**主 Agent 在问用户前**,先扫描用户原始消息,提取以下 7 个维度。能从消息里**明确**提取的,自动填入,不问用户。

| 维度 | 自动判断信号(消息中含以下任一即触发) |
|------|---------------------------------------|
| **需求主题** | 名词短语(审批/CRM/电商/数据看板…) |
| **目标用户** | "C 端/B 端/给客户/给老板/给 XX 部门用" |
| **项目类型** | "全新/新做/做一个" → 全新;"已有 + 加功能/迭代/二期" → 二期 |
| **后端/前端** | "前后端都要/联调" → 后端;"先做前端" → 仅前端 |
| **页面级/功能级** | 含"页面/出页面/UI/界面" → 页面级;含"功能/模块/接口"无"页面" → 功能级 |
| **数据看板** | 含"看板/BI/仪表盘/dashboard/可视化" → 含 BI |
| **行业合规** | 含"金融/医疗/教育/支付/电商/物流/政务" → 强合规;无 → 一般 |

**判定后,主 Agent 输出"画像卡片"**(在第一次问用户时一并展示):
```
📋 需求画像(主 Agent 自动判断)
├── 主题: 兰花品种交易行情(置信度 95%)
├── 用户: C 端兰花爱好者(置信度 80%,待你确认)
├── 项目: 全新(置信度 100%)
├── 模式: 后端同时实现(置信度 70%,因含"销售记录录入")
├── 方案: 页面级(置信度 100%,因含"页面清单")
├── BI: 不含(置信度 90%)
└── 行业: 一般(置信度 100%)
有标 "待你确认" 的项请回复;全部 OK 回复"继续"即可。
```

**Why**: 用户每次答 1-8 个问题,大多数情况答 0-2 个即可,主 Agent 自动判断覆盖 70%+ 场景。

### 必问用户(最多 4 问,通常 1-2 问)

> **触发原则**:主 Agent 已自动判断的维度不重复问;判不准/有歧义/有冲突的才问。

#### 问题 A:需求来源(影响 35 题调研问法)

| 选项 | 场景 | 问法调整 |
|------|------|----------|
| **A. 我就是需求方** | 用户自己有真实需求 | 35 题标准问法,假设用户有完整背景 |
| **B. 转答他人需求** | PM/老板转述,用户不是终端 | 35 题 + 加问"原始需求方是谁/联系方式" |
| **C. 待调研/猜测** | 用户没有真实需求,纯假设/竞品分析 | 35 题改为"假设场景"问法,标"待验证" |

**兜底选项**:"我不确定 → 主 Agent 标注待确认,继续推进"

#### 问题 B:交付范围(后端+页面级+BI 三合一)

| 选项 | 场景 | 触发子技能 / 模板 |
|------|------|------------------|
| **A. 完整模式·页面级·有 BI** | 商业项目 + 含看板 | 16 章 PRD + 数据看板子技能 + 13 子技能全跑 |
| **B. 完整模式·页面级·无 BI** | 商业项目 + 普通 UI | 16 章 PRD + 13 子技能全跑(跳过 BI) |
| **C. 完整模式·功能级·有 BI** | 后台系统 + 含看板 | 6 章 PRD + 数据看板子技能 + 9 子技能 |
| **D. 完整模式·功能级·无 BI** | 后台系统 / 中台模块 | 6 章 PRD + 9 子技能(最常见) |
| **E. 简化模式·页面级·有 BI** | 纯前端 + 有看板 | 16 章 PRD(简化产出) + BI 子技能 |
| **F. 简化模式·页面级·无 BI** | 纯前端 | 16 章 PRD(简化产出) |
| **G. 简化模式·功能级·有 BI** | 纯前端 + 看板 | 6 章 PRD(简化) + BI 子技能 |
| **H. 简化模式·功能级·无 BI** | 纯前端最小 | 6 章 PRD(简化) — 最快 |
| **I. 仅后端 API** | 微服务 / 内部 API | 03 接口 + 12 DB + 13 架构 + 09 后端(无 PRD) |

**Why 合并**:原 4 问(后端/前端、页面/功能、是否 BI、产出范围)中"后端/前端"+"页面/功能"+"是否 BI"是**强相关决策链**,合并成 1 问减少用户认知负担。

**互斥检测**:主 Agent 自动检查 — A/B 选项选了"无 BI"但 Step 1 调研出现"看数据"动词 → 提示用户"是否漏选 BI";A/B 选了"功能级"但用户消息含"页面清单" → 提示"是否需改页面级"。

#### 问题 C:产出范围(4 类受众多选)

| 受众 | 包含角色 | 包含文档(主受众) |
|------|----------|------------------|
| **产品** | PM / 客户 / 对外演示 / 售前 | 01-README / 02-项目整体 / 05-任务 / 06-PRD 主 + 06a-h(8 份) / FigmaMake / 用户调研(共 13 份) |
| **技术** | 前端 / 后端 / 测试 / 运维 / 数据 | 03-接口 / 04-前端指南 / 07-测试 + 07a-c(3 份) / 08-内部交互 / 09-后端 / 10-前端交互 / 11-Mock / 12-DB / 13-架构 / 14-标杆 / 14-技术趋势(共 13 份) |
| **运营** | 行业分析 / 竞品 / 数据看板 / 增长 | 14-行业分析 / 14-竞品分析(共 2 份) |
| **合规** | 法务 / 合规 / DPO | 14-合规清单(共 1 份);**特殊规则**:即使合规被排除,核心 5 条(GDPR / 个保法 / 等保 / 行业资质 / 跨境传输)仍保留为摘要中的 5 个 bullet |

**多选提示**:4 类可多选(1-4 个),不选的受众对应文档降级为 ≤ 200 字摘要(详见 [reference/19](./reference/19-audience-tag-table.md))。

**兜底选项**:"我不确定 → 主 Agent 默认全选(等价 v4.1 行为,适合复杂项目)"

**自动判断**:
- 用户消息中提到具体角色(PM / 工程师 / 法务 / 运营) → 自动勾选对应受众
- 用户说"只关心后端 / 只关心合规审查" → 仅勾选该受众
- 用户说"客户对外演示用" → 仅勾选"产品"
- 都不明示 → 默认全选

#### 问题 D:行业合规与时间(仅当问题 B 选 A/B/C/D 强合规选项时问)

| 合规强度 | 触发词 | 合规章节展开度 |
|----------|--------|----------------|
| **A. 一般** | "通用 SaaS / 内部工具 / C 端产品" | 5 路行业研究中 D 路(合规)简化为 1 段 |
| **B. 行业监管** | "金融 / 医疗 / 教育 / 支付" | D 路全跑 + 加 5 类合规(等保/SOC2/HIPAA/PCI-DSS) |
| **C. 跨境数据** | "海外 / GDPR / 出海 / 双版本" | D 路全跑 + GDPR/CCPA/数据本地化 + i18n 多语言章节 |

**时间预算选项**(新,影响并行度):
- **1-3 天 demo**:3 路行业研究(轻量模式,见 [reference/21](./reference/21-orchestration-lightweight.md)) + 35 题精简为 15 题
- **1-2 周 MVP**:5 路标准 + 35 题标准
- **1-3 月完整**:5 路深度 + 35 题完整 + 用户深度访谈
- **3 月+ 长期**:全模式 + 季度复盘

**Why 加合规与时间**:
- 金融/医疗项目漏合规章节 = 项目上线即违法
- 1 天 demo 项目跑 5 路研究 = 浪费 80% 时间
- 这两个维度漏问,产出 80% 不精准

### 互斥检测清单(主 Agent 必跑)

| 互斥项 | 检测方法 | 处理 |
|--------|----------|------|
| 页面级 + 简化模式 + 仅前端 | 3 选都成立 | 提示"页面级方案建议完整模式,是否切换?" |
| 含 BI + 简化模式 + 功能级 | 3 选都成立 | 提示"BI 看板建议页面级 + 完整模式,是否切换?" |
| 仅技术受众 + 含产品文档 | 互斥 | 提示"选'技术'后 06-PRD 会降级为摘要,PM 同事将看不到完整版" |
| 选合规排除 + 项目含金融/医疗 | 强冲突 | 拒绝继续,提示"金融/医疗项目不能排除合规" |
| 海外/跨境 + 选合规排除 | 强冲突 | 拒绝继续,提示"海外项目必须含 GDPR/数据本地化合规" |

### G0 门控(需求清晰度复述)

Step 0 答完后,**主 Agent 必须**用 G0 复述门控(详见 [reference/20](./reference/20-g0-requirements-clarity.md)):
1. 把用户答案 + 自动判断合并为"最终画像"
2. 检测上述 5 类互斥,如有冲突**主动提示并请求确认**
3. 列出"主 Agent 已为你做的自动判断"(透明化,让用户验证)
4. 用户回"OK"或"修 X 项"后才进 Step 1

**Why**: 实际项目 30% 失败源于"我以为你要 X 但你要 Y"。G0 把答案复述一遍,让用户在 Step 1 大规模启动前发现偏差。

## 执行流程(6 步骨架)

```
       Step 0: 自适应范围确认(主 Agent 静默自动判断 + 4 问用户)
          │ ⚠️ 等用户答(自动判断覆盖 70%+, 通常 1-2 问)
          ▼
       G0 门控: 主 Agent 复述画像 + 互斥检测 + 用户确认
          │ ⚠️ 等用户"OK"或修正
          ▼
       Step 1a: Q1+Q2 关键问题(Skill: openPRD-user-research)
          │ ⚠️ 等用户答
          ├─────────────────────┐
          ▼                     ▼
   Step 1b: Q3-Q35        Step 2: 5 路并行行业研究(或 3 路轻量,见 ref/21)
   (剩余 33 题 / 二期项目 6 题)  A 行业 / B 竞品 / C 标杆
                          D 合规 / E 技术趋势
          │                     │
          └──────────┬──────────┘
                     ▼
       Step 3: 文档产出 5 Track (Skill: openPRD-document-generation)
              · 按 default_audiences 决定"全文 / 200 字摘要"
                     ▼
       Step 4: 四级质量门控 G1 形式 / G1.5 产物 + 抽样 / G2 阶段 / G3 全局
                     ▼
       Step 5: 汇总报告 + 用户验收
                     ▼
       Step 6: 用户复盘(用 18-PLR 模板)→ 收集"哪不准/哪漏"作为下次的输入
```

### ⚠️ 执行规范:必须等待用户回答

**绝对禁止**:询问用户问题后不等待回答就自动继续。

**例外(关键问题先行模式)**:问 Q1+Q2 → 等答 → **立即并行**启动 Step 1b(Q3-Q35)+ Step 2(5 路行业研究)→ 行业研究基于 Q1+Q2 主题即可开始,不需要 Q3-Q5 答案。

### 4 阶段调度(简化版)

完整规则见 [reference/10-orchestration-detail.md](./reference/10-orchestration-detail.md);轻量模式(3 路)见 [reference/21](./reference/21-orchestration-lightweight.md)。

| Stage | 内容 | 类型 |
|-------|------|------|
| **Stage 0** | Step 0 范围(自适应)+ G0 门控 + Step 0.5 Q1+Q2 | 串行·主 Agent |
| **Stage 1** | Step 1b(Q3-Q35 或 6 题二期版)+ 5 路行业研究(或 3 路轻量) | 1+5 并行 |
| **Stage 1.5** | G1 + G1.5 + G2 门控 → 用户确认 | 串行·主 Agent |
| **Stage 2** | 5 Track(A Core/B Interface/C Frontend/D Backend/E Quality) | 并行 |
| **Stage 3** | G1.5 抽样(产物级)+ G3 门控 + 汇总报告 + 验收 + 复盘 | 串行·主 Agent |

**主 Agent 职责**:串行执行用户交互节点 + 调度 subagent + 验证产出 + 整合汇总 + 跑 G0/复盘。

## ⚠️ 页面级方案强制要求(2026-06-03 新增)

> **背景**:用户反馈 openPRD 流程产出的 PRD"太粗、缺少页面级详细描述"。本章节是强制约束,避免再次出现该问题。

| 6 | **信息架构与页面结构** ⭐ | 06c | 页面树/页面清单(含 11 字段)/页面流转图 |
| 7 | **设计系统与组件库** ⭐ | 06d | 基础组件/复合组件/业务组件(每个含变体/状态/交互) |

## 外部能力嵌入(External Capabilities)

> **目的**:openPRD 不是一个孤立的内容生成器,而是**编排各种外部能力**为产出质量服务。

### 能力矩阵

| 能力 | 工具 | 价值 | 失败降级 |
|------|------|------|----------|
| **库文档实时查询** | mcp__context7__resolve-library-id + query-docs | 获取 React/Next.js/Prisma 等库的**最新** API | 退回训练数据 + 标注"未验证版本" |
| **AI 图像/音频/视频** | mcp__minimax__text_to_image/understand_image/text_to_audio/generate_video | 无设计稿时生成示例图、解读用户截图、生成语音示例 | 改用 Mermaid 流程图 + 文字描述 |
| **浏览器自动化** | mcp__chrome-devtools__new_page/take_snapshot/take_screenshot/evaluate_script/lighthouse_audit | 竞品官网抓取、URL 验证、Lighthouse 性能审计、移动端模拟 | 改用 WebFetch + 文字描述 |

### 嵌入原则

1. **不替代核心流程**:外部能力是"加速器",不是"替代品"
2. **明确标注来源**:所有外部能力产出必须标注"由 [工具] 生成"
3. **成本意识**:minimax/text_to_image 等有成本,按需调用
4. **失败兜底**:每个调用都有 try-catch + 降级方案
5. **用户确认**:AI 生成的图/音不直接当作"最终产出",仅作"占位示例"

| 检查项 | 通过标准 |

### 自检输出格式(四级门控)

在向用户汇报前,主 Agent 必须应用**四级质量门控**(详见 [reference/05-quality-gates.md](./reference/05-quality-gates.md) + [reference/13](./reference/13-quality-selfcheck.md) 完整自检清单 + [reference/17](./reference/17-product-level-verification.md) G1.5 产物级验证)。输出格式:

- **G1 单 subagent(形式)**:8 个子技能各自满足章节/方法/来源/数量标准(用户调研 35 题/9 组、行业分析 9 章节/波特五力/PEST、竞品 11 章节/Battle Card、标杆 6 步法/15+ 来源、合规 8 章节/30+ Checklist、技术趋势 7 领域、文档产出 6+1 完整、PRD 拆段 06a-06h/50KB+)
- **G1.5 产物级验证 ⭐**:主 Agent 抽样 3-5 处业务逻辑,扮演新工程师复述实现路径,对照 PRD 验证可行性。输出质疑清单(位置/类型/问题/建议修复/优先级)与用户对齐,P0 项必修。详见 [reference/17](./reference/17-product-level-verification.md)
- **G2 阶段(形式)**:调研/设计/开发/测试 4 阶段各自的完整性(调研=用户+5 路;设计=项目+16 章;开发=接口+DB+架构+任务;测试=P0 100%+边界)。**前置:G1.5 已对齐**
- **G3 全局(形式)**:完整性(N 份/期望 N)、价值密度(5 项交叉)、可执行性(02/06/05 立即评审)、外部能力(context7/minimax/chrome-devtools)。**前置:G1.5 + G2 已通过**
- **5 关键字段交叉**:userId / status / 必填规则 / 错误码 / P0 功能名(完整规则见 [reference/15](./reference/15-five-field-crosscheck.md))
- **结论**:✅ 通过交付 / ⚠️ G1.5 N 项待对齐 / ❌ 需修复 N 项

### 自检不通过时

1. 主 Agent **不得**向用户汇报
2. 修复不通过项(哪个文档缺什么、补什么)
3. 重新自检,全部通过后才汇报

- 用户调研 → 文档产出(不包含代码实现)
- 需求澄清 → Mock 确认 → 方案设计
- **不包含**:直接写代码、修改现有系统、重构

**正确响应**:"这个需求超出了当前 skill 的处理范围(仅限需求到文档的转化)。如需代码实现,请通过常规开发流程处理。"

## Common Mistakes

| 错误 | 修正 |
|------|------|
| 未确认交付范围就开始调研 | 必须先问用户:后端实现 vs 前端Mock |
| 用户说"直接开发"就跳过调研 | 必须先完成调研流程 |
| Mock数据不确认 | 必须与用户确认真实性 |
| 使用近似颜色值 | 必须验证 GlobalStyled.jsx 源码 |
| 文档缺少流程图 | 业务流程必须用流程图展示 |
| 重点内容没有加粗 | 关键内容必须加粗突出 |
| 页面级方案缺 06a-06h 拆分 | 页面级强制 8 段拆分,文档量 ≥ 50KB |
| 跨文档字段不一致 | 5 字段交叉对比必须 100% 一致 |

更完整的常见错误列表见 [reference/16-common-pitfalls.md](./reference/16-common-pitfalls.md)。

## ⚠️ 错误处理与降级策略

**总原则**:**优雅降级 > 硬性中断**。任何外部能力失败都不应阻塞主流程。

### E1:外部能力调用失败

| 工具 | 降级方案 |
|------|----------|
| **context7** | 用训练数据 + 标注"未验证";可改用 WebSearch |
| **minimax** | 用 Mermaid 流程图 + 文字描述代替 |
| **chrome-devtools** | 用 WebFetch 替代;用文字描述代替 |
| **WebSearch / WebFetch** | 全部失败时标注"未找到权威来源",记录 gap |

### E2:用户输入不清晰

业务领域未知 → 默认"通用 SaaS";关键问题拒绝回答 → 礼貌追问最多 2 次仍拒绝则用合理假设 + 标注;35 题部分缺失 → 标记"未回答"继续;信息矛盾 → 列冲突请用户裁决,不裁决取保守值。

### E3:模板/规范冲突

以用户项目实际为准;多份文档字段不一致 → 触发跨文档一致性检查;既有规范 vs 新需求 → 优先遵守既有规范,新需求标"扩展点"。

### E4:技术不可行

用户要求用 A 但 A 不支持 X → 说明限制 + 给 2-3 个备选;性能要求超出架构 → 列瓶颈 + 优化路径(短/中/长期);合规 vs 功能冲突 → 合规优先,标"功能降级"。

### E5:范围蔓延

用户中途加需求 → 礼貌暂停确认是否启动"二期项目";子任务 > 3 个 → 用 TaskList 拆解对齐;文档 > 50 页 → 提示拆分。

### E6:质量门控未通过

G1 不通过 → 不进入 G1.5,先修复;G1.5 质疑未对齐 → 暂停 G2,主 Agent 主动呈交质疑清单与用户对齐(详见 [reference/17](./reference/17-product-level-verification.md));G2 不通过 → 列冲突点要求用户裁决;G3 不通过 → 重新审视范围返工关键文档。

### E7:与用户失联

关键问题等待中 → 礼貌重试 1 次(30min 后);仍无回复 → 用合理默认值 + 标"待确认";文档待验收 → 主动给汇总 + 标"如 24h 无反馈默认通过"。

### E8:技术环境异常

写文件失败 → 检查权限/路径;子 Agent 超时 → 取消用子任务列表分批;网络断开 → 提示用户离线模式;Token 超限 → 自动压缩上下文分段产出。

### 降级策略矩阵

| 工具/能力 | 优先 | 降级 1 | 降级 2 | 降级 3 |
|-----------|------|--------|--------|--------|
| 图像生成 | minimax | 文字 + Mermaid | 文字描述 | 提示用户提供 |
| 图像理解 | minimax understand | 询问用户 | 文字描述 | 跳过 |
| 网页抓取 | chrome-devtools | WebFetch | WebSearch | 提示用户提供 |

| 语音生成 | minimax | 文字脚本 | 跳过 | - |
| 视频生成 | minimax | 截图序列 | 文字脚本 | 跳过 |
