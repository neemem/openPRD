# 文档产出执行步骤详细说明

> 📖 **加载时机**：阅读 openPRD-document-generation 入口时，按需展开以下 6 个步骤的详细说明。

---

## 高价值内容原则

> 🎯 **核心目标**：交付给用户的不是"文档齐全"，而是"拿到就能用"。本章节定义 17 份文档共同的写作准绳。

### ❌ 反模式（堆叠内容）

| 反模式 | 表现 | 后果 |
|--------|------|------|
| 百科式解释 | "什么是 X"、定义、背景 | 用户已知，浪费篇幅 |
| 套话填充 | "建议考虑 X"、"应当注重 Y" | 没有具体方案 |
| 占位空壳 | `[xxx]`、`待定`、`TBD` | 用户拿到无法用 |
| 复制模板 | 章节齐全但全是通用示例 | 与本项目脱节 |
| 罗列清单 | 10 条"注意事项"无优先级 | 无可执行性 |

### ✅ 高价值内容原则

| # | 原则 | 含义 | 检验标准 |
|---|------|------|----------|
| 1 | **回答一个问题** | 每份文档只回答一个核心问题 | 能否用 1 句话说清"这文档解决什么" |
| 2 | **本项目数据** | 用真实业务数据，不用 `xxx` | 数据是否与本项目具体场景对应 |
| 3 | **可执行方案** | 给"怎么做"，不给"考虑什么" | 拿到就能开工/评审/测试 |
| 4 | **明确决策** | 在分歧处给出选择 + 理由 | 不说"二选一"，说"选 A，因为 B" |
| 5 | **跨文档一致** | 接口字段 = DB 字段 = Mock 字段 | 抽 5 个字段交叉对比 |
| 6 | **可视化优先** | 流程/架构用 Mermaid | 1 张图 > 1000 字说明 |
| 7 | **验收可验证** | 每条结论有 Yes/No 判定标准 | 能写"✅/❌"检查项 |

### 17 份文档价值定位速查

| 文档 | 阶段 | 回答的核心问题 | 不回答什么 |
|------|------|----------------|------------|
| 用户调研报告 | 调研 | 用户是谁？痛点？机会？ | 产品形态 |
| 14-行业分析 | 调研 | 行业怎么玩？差异化在哪？ | 具体功能 |
| 02-项目整体 | 设计 | 项目是什么？边界？干系人？ | API/DB/技术选型 |
| 06-PRD | 设计 | 做什么？长什么样？规则？ | 技术实现细节 |
| 08-内部交互 | 设计 | 跨模块/跨服务怎么协作？ | 单模块实现 |
| FigmaMake | 设计 | AI 怎么出设计稿？ | 代码实现 |
| 03-接口文档 | 开发 | API 怎么对？ | 数据库结构 |
| 12-数据库设计 | 开发 | 数据怎么存？ | API 形态 |
| 13-架构设计 | 开发 | 技术怎么选？怎么搭？ | 产品功能 |
| 04-前端指南 | 开发 | 前端怎么实现？ | 后端实现 |
| 09-后端指南 | 开发 | 后端怎么实现？ | 前端实现 |
| 10-前端交互 | 开发 | 状态/异常/空态？ | 后端逻辑 |
| 11-Mock 数据 | 开发 | 前端怎么 mock 跑通？ | 真实联调 |
| 07-测试用例 | 测试 | 怎么验证做对了？ | 业务定义 |
| 05-任务拆分 | 管理 | 谁、什么时间、做什么？ | 怎么做 |
| 01-README | 管理 | 从哪开始？怎么读？ | 具体实现 |

### 价值密度自检（交付前必做）

```markdown
- [ ] 每份文档有 0.0 本文档价值 章节
- [ ] 占位符（xxx/TBD/待定）< 5%
- [ ] 跨文档字段一致（接口/DB/Mock 抽 5 个字段对比）
- [ ] 用户拿到 02/06/05 后能立即评审/排期
- [ ] 价值链清晰：调研→设计→开发→测试→管理
```

## 第二步：代码风格验证

> ⚠️ **必须验证源码，禁止近似值**

### 验证文件清单

| 文件 | 验证内容 | 说明 |
|------|----------|------|
| GlobalStyled.jsx | 主色 #635BFF、成功 #15b79f、危险 #fb5248、文字色 | **必须验证** |
| LayoutStyled.jsx | max-width 1392px、padding 24px | 布局验证 |
| 相关组件 styled 文件 | 表格、表单、空状态样式 | 按需验证 |

### 验证流程

```
1. 尝试读取 src/styles/GlobalStyled.jsx
2. 尝试读取 src/styles/LayoutStyled.jsx
3. 如文件不存在 → 使用模板默认值并在文档中标注"⚠️ 未找到源码，使用默认值"
4. 如文件存在 → 使用实际值，并在文档中标注"✅ 已验证"
```

## 第四步：文档内容规范

### 实用性表现形式

| 形式 | 必须使用场景 | 示例 |
|------|-------------|------|
| **流程图** | 业务流程、状态流转、交互链路 | 用户→前端→后端→数据库 |
| **表格** | 功能列表、参数说明、枚举定义 | API参数表、字段说明 |
| **加粗** | 核心功能、⚠️警告、[必填] | "**[必填]**"、"**⚠️ 禁止**" |
| **代码块** | 请求示例、响应示例 | JSON 格式示例 |
| **有序列表** | 步骤、流程、优先级 | "1. 2. 3." |

### 加粗使用规范

```
【必须加粗的内容】

1. 核心功能 → "**[核心功能]**"
2. 必填字段 → "**[必填]**"
3. ⚠️ 重要提示 → "**⚠️ 重要**"
4. 关键指标 → "**目标：≤500ms**"
5. 验收标准 → "**[验收标准]**"
6. 禁止事项 → "**⚠️ 禁止使用原生 input**"
7. 经验值/估算 → "**⚠️ 经验估算，需调整**"
```

### 流程图使用规范

| 场景 | 推荐形式 | 说明 |
|------|----------|------|
| 业务流程 | ASCII / Mermaid | 用户→前端→后端→数据库 |
| 状态流转 | ASCII 状态机 | 待执行→进行中→已完成 |
| 时序图 | Mermaid sequence | 用户、前端、后端时序 |
| ER图 | Mermaid ER | 表关系 |
| 架构图 | ASCII | 系统组件关系 |

## 第五步：测试用例定制

### 测试用例原则

**根据模块、功能定制，不使用通用模板**

| 模块类型 | 测试重点 | 用例数量参考 |
|----------|----------|--------------|
| 列表模块 | 筛选、排序、分页、批量操作 | 核心10-15条 |
| 表单模块 | 校验、提交、权限、状态回填 | 核心15-20条 |
| 详情模块 | 数据展示、关联加载、状态显示 | 核心8-10条 |
| 流程模块 | 状态流转、节点处理、异常回退 | 核心15-20条 |

### 测试用例结构

每个测试用例必须包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例编号 | 唯一标识 | API-LIST-001 |
| 用例名称 | 简明名称 | 正常分页查询 |
| 前置条件 | 测试前准备 | 已登录，有数据 |
| 测试步骤 | 具体操作 | 1. 调用列表接口 2. page=1 |
| 预期结果 | 期望输出 | 返回 code=200 |
| 优先级 | P0/P1/P2 | P0 |

### 测试用例分类

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| 功能测试 | 验证功能正确性 | 所有功能 |
| 接口测试 | 验证 API 正确性 | 有后端接口 |
| 边界测试 | 验证边界条件 | 输入框、数值 |
| 异常测试 | 验证异常处理 | 网络错误、权限不足 |
| 性能测试 | 验证性能指标 | 列表接口、页面加载 |
