# 质量门控手册（Quality Gates）

> 📖 **使用说明**：本文档是 openPRD 各 subagent 的**质量门控标准**。主 Agent 在调度 subagent 后、汇总报告前，必须按本文档逐项自检。
>
> **Why**：用户原话"需要强力的内容质量及产量"——没有质量门控，文档再多也是空壳。

---

## 1. 门控全景(G1 / G1.5 / G2 / G3)

| 级别 | 时机 | 维度 | 通过标准 | 类型 |
|------|------|------|----------|------|
| **G1 单 subagent 输出** | subagent 完成后立即自检 | 完整性 / 准确性 | 满足"必含项 + 自检清单" | 形式 |
| **G1.5 产物级验证** ⭐ | G1 形式通过后、G2 启动前 | 业务逻辑闭环 / 跨文档流转 / 实现路径自洽 | 主 Agent 抽样复述无信息缺口 + 用户对齐质疑清单 | 产物 |
| **G2 阶段产出** | 每个阶段（调研 / 设计 / 开发）完成后 | 跨文档一致性 / 引用率 | 满足"阶段门控"且 G1.5 已对齐 | 形式 |
| **G3 全局交付** | 最终交付前 | 价值密度 / 可执行性 | 满足"全局门控"且 G1.5 已对齐 | 形式 |

> **形式 vs 产物**:G1/G2/G3 是"形式门控"(机械检查文件/章节/字段),G1.5 是"产物门控"(主 Agent 扮演新工程师抽样复述实现路径)。两者**互补不替代**——形式过≠可落地,产物过才代表能。
>
> **顺序依赖**:G1 → G1.5 → G2 → G3。G1.5 不通过(质疑未对齐)则不进入 G2;G2 不通过则不进入 G3。

## 2. G1：单 subagent 输出门控

### 2.1 通用必含项

每个 subagent 输出必须满足：

| 项 | 检查方法 | 通过标准 |
|----|----------|----------|
| **段落完整** | 抽查段落首尾 | 无"TBD/待补充/TODO"占位符 |
| **数据可追溯** | 抽查数字、引用 | ≥ 80% 数据有来源（数据来源 / 调研 / 用户回答）|
| **Mermaid 图** | 视觉化需求时 | 流程 / 状态 / 时序 / ER 必含 |
| **加粗关键** | 关键发现 / 验收标准 | 关键结论、必填字段、禁止事项已加粗 |
| **代码可执行** | 代码块 | 可直接复制运行的代码示例 |
| **自检清单** | 末尾 | 含"段完成度自检"章节 |

### 2.2 各 subagent 专项门控

| subagent | 必含项 | 自检章节 |
|----------|--------|----------|
| `openPRD-user-research` | ≥ 35 个问题 / 9 组 / 必答标记 | 调研报告末尾"调研结论 + 待确认事项" |
| `openPRD-industry-analysis` | 9 大章节 / 波特五力 / PEST / 行业术语表 | 行业分析报告末尾"段完成度自检" |
| `openPRD-competitor-analysis` | 11 章节 / 波特五力 / SWOT / Battle Card | 竞品分析报告末尾"段完成度自检" |
| `openPRD-deep-research` | 6 步法 / ≥ 15 来源 / 60%+ 12 个月内 | 标杆研究末尾"数据置信度评级" |
| `openPRD-regulation-compliance` | 8 章节 / ≥ 30 项 Checklist / 引用条款 | 合规清单末尾"合规风险评级" |
| `openPRD-tech-trends` | 7 大领域 / 本项目选型建议 | 技术趋势报告末尾"成熟度评级" |
| `openPRD-context7-integration` | 库 ID 来源 / 文档版本 / 代码示例 | 引用块末尾"来源标注" |
| `openPRD-minimax-integration` | Prompt 模板 / 失败降级 / 成本警告 | 调用块末尾"成本警告" |
| `openPRD-chrome-devtools-integration` | 资产路径 / 性能审计 / 资源释放 | 调用块末尾"资源释放" |
| `openPRD-document-generation` | 6+1 文档完整 / 跨文档字段一致 | 各文档末尾"自检清单" |

### 2.3 失败处理

| 失败类型 | 处理 |
|----------|------|
| 占位符 > 20% | 立即修复，标记为"草稿" |
| 数据无来源 | 立即补来源或删除 |
| Mermaid 图错误 | 立即修复（语法问题） |
| 跨文档字段不一致 | 主 Agent 协调修复 |

## 2.5 G1.5：产物级验证门控 ⭐(新增)

> 完整规则、抽样模板、案例详见 [reference/17-product-level-verification.md](./17-product-level-verification.md)。本节列出**触发条件 + 抽样规则速查**。

### 2.5.1 为什么需要 G1.5

G1 形式门控查"文件存在、章节齐全、字段齐备",但 PRD 写完不等于能落地。例:06-PRD §3.2 US-05 "用户取消订单"——G1 看到章节存在即通过,G1.5 会问"取消后支付如何退款?"——这是 G1 查不出的设计缺口。G1.5 是主 Agent 扮演"新来的工程师",抽样验证产物是否真的能跑通。

### 2.5.2 触发条件 + 抽样规则

| 模式 | 是否必走 | 抽样数 |
|------|----------|--------|
| 完整模式 - 页面级 | ✅ **必走** | 核心 5 处 / 关键 3 处 / 辅助 1-2 处 |
| 完整模式 - 功能级 | ✅ 推荐走 | 核心 3 处 / 关键 2 处 |
| 简化模式 - 页面级 | ✅ **必走** | 06 抽 5 处 / 03/12 抽 3 处 |
| 简化模式 - 功能级 | ⚠️ 推荐走 | 06 抽 3 处 |

每处抽样必含 3 类:
1. **端到端实现路径**:1 个 P0 US 从 UI → 接口 → DB → Mock 是否信息自洽
2. **状态转换闭环**:1 个核心业务规则的状态机是否含全部 入态/出态/异常态
3. **跨文档字段流转**:1 个字段在 06-PRD/03-接口/12-DB/11-Mock 4 处是否一致

### 2.5.3 验证方法(主 Agent 步骤)

1. 选 1 个 P0 US,**不**复述,问:"如果我是新来的工程师,看完这 US 我会怎么实现?"
2. 走查 06-PRD → 03-接口 → 12-DB → 11-Mock,标"信息缺口"
3. 输出"质疑清单"(**不输出**"通过/不通过"),交用户对齐

### 2.5.4 输出格式(质疑清单)

| # | 位置 | 类型 | 问题 | 建议修复 | 优先级 |
|---|------|------|------|----------|--------|
| 1 | 06-PRD §3.2 US-05 | US 未覆盖异常 | "用户取消订单"未定义"取消后支付是否退款" | 补 US-05 扩展流;或加 /refund 接口 | P0 |
| 2 | 12-DB `approvals` 表 | 字段缺失 | US-03 提到"审批历史"但表无 `approval_log` | 新增 `approval_log` 表 | P0 |
| 3 | 03-API `POST /approvals` | 跨文档不一致 | 06-PRD `approvers` 是 `bigint[]`,03-API 是 `number[]` | 统一类型 + 加映射说明 | P1 |

### 2.5.5 与 G1/G2/G3 的关系

```
G1 形式过 → G1.5 产物验证 → 用户对齐 → G2 阶段过 → G3 全局过 → 交付
                                  ↓
                              设计冲突 → 暂停
```

- G1 → G1.5:G1 形式通过后才启动 G1.5(无基础结构则无法抽样)
- G1.5 → G2:**G1.5 质疑必须与用户对齐后才进入 G2**(否则下游基于错误前提开发)
- G3:确认 G1.5 已通过

> **关键**:G1.5 不通过**不会硬性失败**,而是"产出质疑清单返回用户对齐"。这与 G1/G2/G3 的硬性 ✅/❌ 形成互补——形式可机械化检查,产物必须人工对齐。失败处理详见 §6.3。

## 3. G2：阶段产出门控

> **前置条件**:G1 形式门控 ✅ + G1.5 产物级验证已与用户对齐(详见 [reference/17](./17-product-level-verification.md))。G1.5 质疑清单未对齐则**不进入 G2**。

### 3.1 调研阶段门控

| 项 | 通过标准 |
|----|----------|
| 用户调研 | 35 题回答 ≥ 80% + Q33 复述确认 + Q35 下一步确认 |
| 行业研究 5 路 | 5 份独立报告 + 1 份综合洞察 + 冲突解决 |
| 综合置信度 | ≥ 70% 关键结论有数据/引用支撑 |
| 输出物 | 用户调研报告 + 14-行业分析报告 + 14-竞品分析报告 + 14-标杆研究 + 14-合规清单 + 14-技术趋势 |

### 3.2 设计阶段门控

| 项 | 通过标准 |
|----|----------|
| 项目整体说明 | 5W 齐全 + 商业模式画布 + 价值主张 |
| PRD 06 主控 | 16 章索引 + 段间契约清晰 |
| PRD 06a-06h | 各段完成度自检 ≥ 90% |
| 06-PRD 文档量 | ≥ 50KB（页面级） / ≥ 15KB（功能级） |
| 跨文档字段一致 | 用户调研 / 行业分析 / PRD 中关键字段名一致 |

### 3.3 开发阶段门控

| 项 | 通过标准 |
|----|----------|
| 接口文档 | ≥ 80% 核心 API 有完整定义 |
| 数据库设计 | 实体字段与接口字段一致 |
| 架构设计 | 7 大领域（前端/后端/数据/AI/安全/通信/国产化）齐全 |
| Mock 数据 | 字段类型与接口一致 |
| 任务拆分 | 每个任务 1-3 天 + 验收标准 |

### 3.4 测试阶段门控

| 项 | 通过标准 |
|----|----------|
| 测试用例 | P0 覆盖 100% + 边界异常 + 性能 |
| 验收标准 | 来自 PRD AC + Given-When-Then |

## 4. G3：全局交付门控

> **前置条件**:G1 ✅ + G1.5 ✅(质疑清单已对齐 + P0 项已修复)+ G2 ✅。任一不通过则不启动 G3。
> 详细 G1.5 规则见 [reference/17-product-level-verification.md](./17-product-level-verification.md)。

### 4.1 完整性

| 检查项 | 通过标准 | 验证方法 |
|--------|----------|----------|
| 文档数量 | 完整模式 14 份 / 简化模式 9 份 | 对照清单逐项打勾（详见 [reference/14-mode-tag-table.md](./14-mode-tag-table.md)） |
| 模板应用 | 06-PRD 拆 8 段（页面级）/ 单文档（功能级） | 检查 06a-06h 存在性 |
| 子技能产出 | 所有调用的 subagent 有产物 | 检查 `*-docs/` 目录 |
| 跨文档引用 | 上下游有引用链接 | 抽查 10 个引用 |

### 4.2 价值密度

| 检查项 | 通过标准 |
|--------|----------|
| 每份文档有"回答的核心问题" | 17/17 明确 1 句话 |
| 占比空壳 | < 5%（TBD/待补充/根据需求定）|
| 跨文档字段一致 | 5 个关键字段（userId/status/必填规则/错误码/P0 功能名）— 详见 [reference/15-five-field-crosscheck.md](./15-five-field-crosscheck.md) |
| 文档量级 | 06-PRD ≥ 50KB（页面级）/ 14-行业分析 ≥ 30KB / 02-项目整体 ≥ 20KB |
| 可执行性 | 用户拿到 02/06/05/06d 后能立即评审/排期 |
| 价值链清晰 | 调研→设计→开发→测试→管理 上下游明确 |
| 跨文档研究严谨性 | 14-行业分析 ≥ 15 来源且 60%+ 12 个月内 / 用户调研 ≥ 2 种方法 + ≥ 5 个用户引用 |
| 外部能力使用 | 涉及具体技术栈时调用 context7 / 涉及视觉化时调用 minimax / 涉及实时数据时调用 chrome-devtools |

### 4.3 5 个关键字段交叉对比（必查）

> 详细规则、验证命令、修复流程详见 [reference/15-five-field-crosscheck.md](./15-five-field-crosscheck.md)。
> 下表为速查版，**主表以 reference/15 为准**。

| 字段 | 出现位置 | 验证方法 |
|------|----------|----------|
| 字段名（如 `userId`）| 06-PRD / 03-接口 / 12-DB / 11-Mock | 4 处拼写完全一致 |
| 枚举值（如 `status`）| 06-PRD / 03-接口 / 11-Mock / 07-测试 | 4 处枚举值完全一致 |
| 必填规则 | 06-PRD / 03-接口 | 必填标记一致 |
| 错误码 | 03-接口 / 09-后端 / 07-测试 | 错误码与测试用例对应 |
| P0 功能名 | 06-PRD / 07-测试 / 05-任务 | P0 功能名在三处一致 |

### 4.4 页面级专项门控

| 项 | 通过标准 |
|----|----------|
| 页面清单 | 所有页面 11 字段齐全（名称/路径/类型/是否需登录/页面状态/SEO/分享/相关US/关联UC/预加载/权限）|
| 组件库 | 每个基础组件 5 维度（变体/尺寸/状态/交互/可访问性）|
| 交互矩阵 | 每个页面 8 状态（初始/加载/空数据/错误/网络错误/服务器错误/无权限/成功）|
| 数据模型 | ERD + 实体字段 + 索引 |
| 接口契约 | 请求/响应/错误码/超时/重试/幂等 |
| 页面流转图 | Mermaid，含正常/异常分支 |
| 业务规则 | 每条规则编号 + 触发 + 逻辑 + 输出 + 异常 |
| 状态机 | 核心对象（订单/客户/支付）含状态枚举 + 转移 |
| 埋点 | 每个 P0 功能含事件设计 |
| 测试用例 | 每个 P0 US 含 Given-When-Then |

## 5. 自检输出格式

主 Agent 在向用户汇报前输出：

> 完整自检清单/通过标准/验证方法详见 [reference/13-quality-selfcheck.md](./13-quality-selfcheck.md)。

### G1.5 产物级验证 ⭐(详见 reference/17)
- [✅/⚠️/❌] 06-PRD 抽样 5 处:质疑清单 N 项(P0×a / P1×b / P2×c),用户对齐情况

### G2 阶段门控(前置:G1.5 已通过)
- [✅/❌] 调研阶段：用户调研 + 5 路行业研究

- [✅/❌] 完整性：N 份文档 / 期望 N
- [✅/❌] 价值密度：5 项交叉对比
- [✅/❌] 可执行性：02/06/05 立即可评审
- [✅/❌] 外部能力：context7 / minimax / chrome-devtools

## 6. 修复流程

### 6.1 不通过时

1. 主 Agent **不得**向用户汇报
2. 修复不通过项（哪个 subagent / 哪个文档 / 缺什么）
3. 重新调用该 subagent
4. 重新自检
5. 全部通过后才汇报

### 6.2 修复优先级

| 优先级 | 类型 | 修复策略 |
|--------|------|----------|
| P0 | 文档缺失 / 关键字段不一致 / G1.5 P0 质疑未修 | 立即补全 |
| P1 | 数据无来源 / 自检清单缺失 / G1.5 P1 质疑未对齐 | 补全 + 标注 |
| P2 | 视觉化不足 / 价值密度偏低 / G1.5 P2 质疑(Mock 不真实等) | 增加图表 / 引用 |
| P3 | 格式细节 | 批量优化 |

### 6.3 G1.5 失败专项处理(详见 [reference/17](./17-product-level-verification.md))

| 失败类型 | 处理路径 |
|----------|----------|
| G1.5 质疑清单未与用户对齐 | 暂停 G2,主 Agent 主动向用户呈交质疑清单要求对齐 |
| 用户接受质疑清单 | 按质疑修设计/补 US → 重跑 G1.5 → 通过后进入 G2 |
| 用户认为不是问题 | 在 PRD 加备注"已与用户确认 XX 不需补充" → 进入 G2 |
| 用户与设计冲突(不可调和) | 暂停交付,标"设计冲突"返回用户重新决策范围 |

> **关键**:G1.5 失败**不会硬性 ❌**,而是"产出质疑清单返回用户对齐"。这是与 G1/G2/G3 的最大区别——形式门控可机械化,产物门控必须人工对齐。

## 7. 用户验收

主 Agent 完成所有自检后，**主动邀请用户验收**：
