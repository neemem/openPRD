# 19 · 受众标签表（Audience Tag Table）

> 📖 **使用说明**：本文档是 openPRD **"产出范围可选化"功能**的**唯一来源**——4 类粗粒度受众(产品/技术/运营/合规) × 33 份模板 的映射表 + 摘要降级规则 + 调度逻辑。
> 引用方:主 SKILL.md "问题四" / `openPRD-document-generation` Step 3 调度 / `.openprd/config.json` 的 `default_audiences` 字段。
>
> **Why**:v4.2 之前,openPRD 默认全产出 16 份文档 + 5 份行业研究。但实际项目中不同角色的关注点差异巨大:产品经理不需要 03-接口文档的完整 DDL,后端工程师也不需要 14-行业分析的 9 章节。**产出范围可选化**让用户在 Step 0 选 4 类受众,被排除的受众对应文档降级为 ≤ 200 字摘要,既保留"全产出"的完整性,又避免"信息过载"。

---

## 0. 文档目的

- **统一受众定义**:4 类受众(产品/技术/运营/合规)的角色边界、关心内容、单独成类理由
- **单一映射表**:33 份模板 × 4 类受众的"主受众 / 次要受众"标签
- **可机械化验证**:每个模板给出"摘要降级规则"和"提取算法",支持自动化
- **调度规则**:Step 3 据 `default_audiences` 决定"产出全文"还是"产出 200 字摘要"

---

## 1. 4 类受众定义

### 1.1 产品(PM / 客户 / 对外演示)

**包含角色**:
- **产品经理(PM)**:负责产品规划、需求设计、跨部门协调
- **客户 / 业务方**:B 端项目中的需求方、对接人、决策者
- **对外演示者**:售前、销售、解决方案专家,需要"一页纸"给客户讲清楚

**关心什么**:
- "做什么、为谁做、价值是什么"(Why / What)
- 用户画像、用户旅程、用户故事(Given-When-Then)
- 页面结构、组件库、交互矩阵(页面级方案)
- 业务规则、状态机、决策表
- 验收标准、P0 功能定义
- 行业趋势、竞品对标(高层级)

**为何单独成类**:
- 产品的"语言"是"用户故事 + 验收标准",不是"DDL + API"
- 对外演示时不能给客户看 03-接口文档的完整 JSON Schema
- 13 份模板(01/02/05/06/06a-h/08/FigmaMake/用户调研)直接服务产品

### 1.2 技术(前端 / 后端 / 测试 / 运维 / 数据)

**包含角色**:
- **前端工程师**:React/Vue/小程序/H5 等客户端实现者
- **后端工程师**:API 设计、数据库、缓存、消息队列实现者
- **测试工程师(QA)**:功能/E2E/边界/性能测试用例设计者
- **运维 / SRE**:部署、监控、灰度、回滚负责人
- **数据工程师**:数据仓库、ETL、BI 报表实施者

**关心什么**:
- "怎么实现、字段怎么定、API 怎么走"(How)
- 接口契约(请求/响应/错误码/分页/幂等)
- 数据库表结构、索引、约束、ERD
- 架构图、技术选型、部署方案
- 测试用例、覆盖率、边界条件
- 性能指标(QPS/P99/并发)、监控告警
- 行业技术趋势、标杆架构、开源库选型

**为何单独成类**:
- 技术的"语言"是"代码 + 配置 + DDL",不是"用户故事"
- 9 份模板(03/04/07/07a-c/08/09/10/11/12/13)直接服务技术
- 14-技术趋势 + 14-标杆研究 主要是技术决策依据

### 1.3 运营(行业 / 竞品 / 数据看板 / 增长)

**包含角色**:
- **行业分析师**:市场研究、规模测算、PEST 分析
- **竞品分析师**:对标分析、波特五力、SWOT、Battle Card
- **数据分析师(BI)**:数据看板设计、KPI 拆解、数据驱动决策
- **增长负责人(增长 / 用户运营)**:AARRR 漏斗、留存、转化、ROI

**关心什么**:
- "市场多大、对手是谁、机会在哪"(Market / Competition / Growth)
- 行业特点、产业链、术语、规模
- 竞品功能矩阵、商业模式、定价
- 标杆研究、最佳实践
- 数据看板、KPI 监控、归因分析
- 增长漏斗、A/B 测试、用户分群

**为何单独成类**:
- 运营的"语言"是"数据 + 趋势 + 策略",不是"产品需求"
- 2 份核心模板(14-行业分析、14-竞品分析)直接服务运营
- 14-标杆研究的"产品对标"维度次要服务产品(对标客户体验)
- openPRD-data-dashboard 子技能的 dashboard-self-check 文档是"运营"的强相关

### 1.4 合规(法务 / 合规)

**包含角色**:
- **法务(Legal)**:合同审核、法律风险评估
- **合规专员(Compliance)**:GDPR / 个人信息保护法 / 行业监管要求
- **数据保护官(DPO)**:数据合规、隐私影响评估

**关心什么**:
- "什么不能做、什么必须披露"(What Not To Do)
- 数据合规清单(GDPR / CCPA / 个人信息保护法 / 行业法规)
- 隐私政策、用户协议、Cookie 政策
- 跨境数据传输、数据本地化
- 反洗钱(AML)、反恐融资(CFT)(金融/支付行业)
- 行业资质、认证(等保 / SOC2 / ISO27001)

**为何单独成类**:
- 合规的"语言"是"条款 + 清单 + 风险等级",不是"功能"
- 1 份核心模板(14-合规清单)直接服务合规
- 合规清单若被排除,可能遗漏关键法规风险,导致法律纠纷
- 重要约束:即使运营/技术受众均被排除,**合规文档的"核心 5 条"必须保留为摘要**(见 §3 摘要降级规则)

## 3. 摘要降级规则(关键)

### 3.1 200 字摘要的固定结构

每个被排除受众对应的文档,Step 3 调度时**降级为 200 字摘要**,摘要必须包含两部分:

```
[核心决策 3-5 个 bullet,每个 ≤ 50 字]
+
[关键数字/对象](如 P0 功能数、接口数、表数、合规条款数)
```

**示例(以 `templates/03-接口文档.md` 降级为例)**:

```markdown
## 摘要(降级输出,200 字内)

> 本文档已降级。原因:运营/合规未列入产出范围,接口完整定义仅对技术受众开放。
> 以下为运营/合规视角需关注的核心要点。

- **接口总数**:42 个 RESTful API(用户 8 / 订单 12 / 支付 6 / 其他 16)
- **核心 P0 接口**:用户登录、订单创建、支付下单(3 个,均需防重放/幂等)
- **错误码体系**:6 位数字 AABBCC,跨 5 模块(11/22/33/44/55),共 38 个错误码
- **数据合规**:涉及个人信息的接口 18 个,均需 GDPR / 个保法合规审查(详见 14-合规清单)
- **性能指标**:核心接口 P99 ≤ 200ms,支付接口 ≤ 500ms,QPS 峰值 1000

**关键对象**:`userId`(UUID v4)/ `status`(PENDING/ACTIVE/DISABLED/DELETED)
**完整版见**:`templates/03-接口文档.md`(技术受众可访问)
```

### 3.2 摘要提取算法(自动化)

主 Agent 在 Step 3 调用 `openPRD-document-generation` 时,对每个被排除受众的文档,执行以下提取逻辑:

```python
def extract_summary(template_path: str, excluded_audiences: list) -> str:
    """
    从模板中提取 ≤ 200 字摘要,供被排除受众查看
    """
    content = read_file(template_path)

    # 1. 提取 H1 标题(模板名)
    h1 = extract_h1(content)

    # 2. 提取第一个 ### 章节(模板说明)
    first_section = extract_first_h3(content)

    # 3. 提取"⭐ 决策"标注(v4.0 新增)
    decisions = extract_decisions(content)  # 最多 5 条

    # 4. 提取关键数字(接口数 / 表数 / P0 数)
    numbers = extract_key_numbers(content)

    # 5. 组装 200 字摘要
    summary = f"## {h1} 摘要(降级输出,200 字内)\n\n"
    summary += f"> 本文档已降级。受众{', '.join(excluded_audiences)}未列入产出范围。\n\n"
    summary += "**核心决策**:\n"
    for d in decisions[:5]:
        summary += f"- {d}\n"
    summary += f"\n**关键数字/对象**:{numbers}\n"
    summary += f"\n**完整版见**:`{template_path}`(主受众可访问)\n"

    # 6. 截断至 200 字(中文按字符,英文按词)
    return truncate_to_200_chars(summary)
```

### 3.3 Fallback(若模板无内容)

若模板在 v4.2.1 阶段尚未产出实际内容(仅是空模板),使用以下 fallback:

```markdown
## 摘要(降级输出,200 字内)

> 本文档未产出完整内容(模板预留,v4.3 起填充)。

- **模板定位**:[模板的 1 句话描述,来自 INDEX.md]
- **主受众**:**[产品/技术/运营/合规]**
- **次要受众**:[产品/技术/运营/合规,可多选]
- **核心产出**:[1-2 句话说明该文档的核心价值]
- **降级原因**:[产品/技术/运营/合规]未列入本次产出范围

**完整版见**:`templates/XX-XXXX.md`(主受众可访问)
```

### 3.4 摘要占位规范(模板内嵌)

为支持 v4.2.2 的自动化提取,每份模板**末尾**应预留 `## 摘要(降级输出,200 字内)` 段(空内容即可,v4.2.1 阶段先保留段标题):

```markdown
## 摘要(降级输出,200 字内)

<!--
本段由 openPRD-document-generation 在 v4.2.2+ 自动填充。
填充逻辑:从 H1 + 第一个 ### + ⭐ 决策标注 提取前 200 字。
当前为占位段,v4.2.1 阶段保持空内容。
-->
```

> **注**:v4.2.1 阶段**不**实际修改 templates/*.md(硬约束"33 个现有模板不触动"),仅在 reference/19 文档中说明"摘要占位规范",v4.2.2+ 再由 `openPRD-document-generation` 自动补段。

### 3.5 特殊规则:合规章节降级时必须保留"核心 5 条"

即使合规受众被排除,**14-合规清单模板.md 的"核心 5 条"(GDPR / 个保法 / 等保 / 行业资质 / 跨境传输)必须保留为摘要中的 5 个 bullet**,不进入降级队列。

**Why**:合规风险是"零容忍"——遗漏 1 条可能导致法律纠纷或下架。技术 / 产品 / 运营均需知晓"本项目涉及哪些合规底线"。

## 4. Step 3 调度逻辑

### 4.1 调度规则总表

主 Agent 在 `openPRD-document-generation` Step 3 阶段,按以下规则调度:

| 受众匹配情况 | 调度动作 | 产出物 |
|--------------|----------|--------|
| **受众匹配**(模板主受众 ∈ 用户的 `default_audiences`) | 正常产出 | 完整文档(按现有模板规范) |
| **受众不匹配**(模板主受众 ∉ 用户的 `default_audiences`) | 降级为摘要 | ≤ 200 字摘要(§3.1 固定结构) |
| **次要受众匹配**(模板次要受众 ∈ 用户的 `default_audiences`) | 主受众产出全文,次要受众在文末追加"对[受众]的提示"段 | 完整文档 + 次要受众提示段 |
| **特殊规则**:合规章节被排除 | 仍保留"核心 5 条"为摘要中的 5 个 bullet(§3.5) | 摘要 + 核心 5 条 |

## 5. 与 .openprd/config.json 集成

### 5.1 新增字段:`default_audiences`

**位置**:`.openprd/config.json` 顶层(与 `default_mode` / `default_tier` 同级)

**类型**:`array[string]`,元素为 `"产品" / "技术" / "运营" / "合规"` 之一

**默认值**:`["产品", "技术", "运营", "合规"]`(全选,等价于 v4.1 行为)

**示例**:

```json
{
  "default_mode": "complete",
  "default_tier": "完整",
  "default_audiences": ["产品", "技术", "运营", "合规"],
  "user_preferences": {
    "backend_implementation": "",
    "page_level": "",
    "industry": "",
    "target_users": ""
  },
  "history_path": "./.openprd/history/"
}
```

### 5.2 兼容老用户(无该字段时全选)

**规则**:若 `.openprd/config.json` 中**无** `default_audiences` 字段,主 Agent **视为全选**(等价于默认值),不报错。

**Why**:v4.2 之前的老用户配置文件没有该字段,直接报错会导致**全量老用户**无法使用 openPRD。Step 0 第四问(主 SKILL.md 新增)会主动询问用户,允许他们更新配置。

### 5.3 字段读取逻辑

主 Agent 启动时:

```python
# 伪代码
if "default_audiences" not in config:
    default_audiences = ["产品", "技术", "运营", "合规"]  # 兼容老用户
else:
    default_audiences = config["default_audiences"]
    # 校验值是否在 4 类中
    valid_audiences = {"产品", "技术", "运营", "合规"}
    for a in default_audiences:
        if a not in valid_audiences:
            raise ValueError(f"Invalid audience: {a}. Must be one of {valid_audiences}")
```

## 6. 与 openPRD 其他子技能协作

### 6.1 协作矩阵

| 子技能 | 协作点 | 触发条件 |
|--------|--------|----------|
| `openPRD-document-generation` | Step 3 调度时**读取** `default_audiences`,对每个模板决定"全文/摘要" | 总是触发(Step 3 必须) |
| `openPRD-data-dashboard` | dashboard-self-check 文档的"主受众"是"运营",若运营被排除则降级 | 用户选"产品/技术"但**触发了数据看板**时 |
| `openPRD-user-research` | 用户调研报告的主受众是"产品",35 题全部问完才生成报告;若产品被排除,报告降级 | 用户选了"技术/运营/合规"为主 |
| `openPRD-industry-analysis` | 14-行业分析的主受众是"运营",若运营被排除,行业分析降级 | 用户选了"产品/技术/合规"为主 |
| `openPRD-competitor-analysis` | 14-竞品分析的主受众是"运营",若运营被排除,竞品分析降级 | 同上 |
| `openPRD-regulation-compliance` | 14-合规清单的主受众是"合规",若合规被排除**但**仍保留"核心 5 条"为摘要 | 用户选了"产品/技术/运营"为主 |
| `openPRD-context7-integration` | 库文档查询不受受众影响(技术主) | 无变化 |
| `openPRD-minimax-integration` | AI 图像/视频生成不受受众影响(产品主) | 无变化 |
| `openPRD-chrome-devtools-integration` | 浏览器自动化不受受众影响(技术主) | 无变化 |

### 6.2 openPRD-data-dashboard 特殊处理

**场景**:用户选 default_audiences = ["产品", "技术"](无运营),但需求含数据看板(Step 0 第三问触发)。

**处理**:
1. `openPRD-data-dashboard` 子技能仍正常触发(因为含数据看板词)
2. dashboard-self-check 文档的"主受众"是"运营",但运营被排除 → 降级为 200 字摘要
3. 摘要内容:7 原则清单 + 6 段 layout 骨架列表(避免运营/管理层完全看不到数据看板设计原则)

**Why**:数据看板的核心受众虽是运营,但产品和技术(尤其是产品)也需要知晓"7 原则",否则做出来的 dashboard 管理层看不懂。

## 8. 常见误用

| 误用 | 后果 | 修正 |
|------|------|------|
| 用户说"我只关心后端",default_audiences = ["技术"] | 产品文档全部降级为 200 字摘要,后端无法理解"做什么、为什么" | 至少保留"产品"+"技术",让后端看到 06-PRD 核心决策 |
| 合规章节被完全排除(连核心 5 条都没有) | 遗漏 GDPR / 个保法关键条款 | §3.5 特殊规则:即使合规受众被排除,核心 5 条必须保留 |
| 摘要超 200 字 | 失去"降级"意义,信息仍然过载 | 严格按 §3.1 固定结构 + §3.2 截断算法 |
| 摘要只有"模板描述"没有"核心决策" | 摘要成了"目录",用户得不到任何新信息 | 摘要必须含 3-5 个 ⭐ 决策 bullet |
| 误用 `default_audiences` 字段做"权限控制" | 摘要阶段无权限,用户看不到完整内容 | 摘要阶段所有受众均可访问完整文档链接(只是默认不展开) |
| 受众数选 0(空数组) | 所有文档全部降级为摘要,等于"没产出" | 主 Agent 启动时校验,空数组视为全选 |
| 误以为"运营 = 数据看板" | 排除运营后,数据看板相关文档全降级 | 实际:数据看板主受众是运营,但 7 原则摘要仍对产品/技术可见 |
