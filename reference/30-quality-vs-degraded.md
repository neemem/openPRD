## 2. 为什么会"质量不变但速度变慢"?

### 2.1 内容质量的保证机制

**质量来自方法论,不是工具**:

| 质量来源 | 是否依赖子 Agent? | 降级影响? |
|---|---|---|
| **35 题调研** | ❌ 否(主 Agent 直接问) | ✅ 零影响 |
| **5 路行业研究** | ✅ 是(并行加速) | ⚠️ 时间变长,但内容质量不变 |
| **文档产出** | ⚠️ 部分(主 Agent 串行写) | ⚠️ 速度慢,但内容深度不变 |
| **质量门控** | ❌ 否(主 Agent 跑) | ✅ 零影响 |
| **置信度评分** | ❌ 否(主 Agent 评分) | ✅ 零影响 |
| **跨文档一致性** | ❌ 否(主 Agent 检查) | ✅ 零影响 |

**结论**:**质量来自方法论,方法论在主 Agent**。子 Agent 只是**加速器**,不是**质量保证器**。

### 2.2 速度影响的具体原因

**5 路并行 vs 1 路串行**:
- Claude Code:5 个 SubAgent 并行处理 5 份研究 → 3 min
- OpenCode:1-2 个 SubAgent + 主 Agent → 100 min
- Codex:无 SubAgent,主 Agent 串行 → 180 min

**数学解释**:
- 5 份研究内容相近(各 5K 字)
- 串行时间 ≈ 并行时间 × 5 = 15 min(单次) vs 3 min(5 次并行)
- 加上其他环节,总时间 75min → 180min(+140%)

### 2.3 速度慢 ≠ 质量差

**类比**:
- 🚗 跑车 vs 卡车:都能到目的地,只是速度不同
- 📞 视频会议 vs 邮件:能传达信息,只是效率不同
- 🍳 大厨 vs 家庭主妇:能做好菜,只是效率不同

**openPRD V5.0 降级**:
- 主 Agent 像经验丰富的"大厨",能用单线程做出完整产品
- 5 个 SubAgent 像 5 个"厨师",并行出菜但每个都是同水平的厨师
- 质量由"大厨/厨师"水平决定,不依赖并行数量

## 3. 真实风险与缓解

### 3.1 主要风险

| 风险 | 影响 | 严重度 | 缓解 |
|---|---|---|---|
| **上下文窗口溢出** | 主 Agent 处理 38 份文档时,Token 可能超过模型限制 | 中 | 增量处理(每份完成即总结) |
| **单 Agent 失败 = 全部失败** | 5 路并行是 5 份文档,失败只影响 1 份;串行失败则全部 | 高 | 主 Agent 自动重试 + checkpoint |
| **Token 成本** | 单 Agent 长上下文消耗更多 Token | 中 | 增量处理 + 上下文压缩 |
| **质量一致性** | 单一 Agent 可能风格不统一 | 低 | 模板 + 统一规范 |
| **维护困难** | 单文件版本难以增量更新 | 中 | distribute.sh 可重新生成 |

### 3.2 Codex 特殊风险

Codex CLI **无子 Agent** 是最大风险:
- ❌ **不能并行**:所有研究/产出都是串行
- ❌ **上下文压力**:所有 38 份文档都在一个上下文
- ❌ **失败影响大**:一个错误可能影响所有内容

**缓解**:
- ✅ 主 Agent 自动增量处理(每完成一份,刷新上下文)
- ✅ 强 checkpoint 机制
- ✅ 强化的质量门控

### 3.3 OpenCode 中等风险

OpenCode **部分子 Agent**:
- ⚠️ **能力有限**:子 Agent 可能不可用
- ⚠️ **降级到单 Agent**:类似 Codex

**缓解**:
- ✅ 主 Agent 自动检测并降级
- ✅ 单 Agent 串行完成

## 5. 质量保证的额外机制

V5.0 设计了**多层质量保证**,不依赖单一特性:

### 5.1 4 级质量门控(降级后仍跑)

```
G1 形式 → G1.5 产物 → G2 阶段 → G3 全局
  ↓         ↓           ↓         ↓
  全部由主 Agent 执行,不需要子 Agent
```

### 5.2 100 分制置信度评分(降级后仍跑)

```
引用量(40) + 引用依据专业性(35) + 是否有背书(25) = 100 分
  ↓
由主 Agent 自动评估,不依赖子 Agent
```

### 5.3 5 字段交叉验证(降级后仍跑)

```
userId / status / 必填规则 / 错误码 / P0 功能名
  ↓
主 Agent 自动比对,不依赖子 Agent
```

### 5.4 35 题决策 + 14 ADR(降级后仍跑)

```
35 题决策 + 14 个 ADR = 跨文档一致性保障
  ↓
这些规则由主 Agent 强制执行,不依赖子 Agent
```

**结论**:**质量保证机制不依赖子 Agent,降级不影响质量**。

## 7. 透明总结

### 7.1 直接回答用户问题

> **"降级会影响质量吗?"**

**答**:
- ✅ **内容质量**:零影响(方法论/门控/内容深度全一致)
- ⚠️ **产出效率**:影响 30-140%(Codex 最慢)
- ⚠️ **架构能力**:子 Agent 不可用时丧失
- ⚠️ **维护性**:单文件模式不利于增量更新

### 7.2 openPRD V5.0 的承诺

✅ **承诺 1**:**所有平台产出内容质量 100% 一致**(方法论 + 门控 + 深度)

✅ **承诺 2**:**Claude Code CLI 是主路径**(90% 用户推荐)

✅ **承诺 3**:**降级是"能力降级"而非"质量降级"**(内容由主 Agent 保证)

✅ **承诺 4**:**用户自由选择平台**(transparent disclosure)

### 7.3 给用户的建议

**如果项目重要**:
- **必须用 Claude Code CLI**(完整能力)
- 不要再纠结"多平台",直接用最稳的

**如果项目演示**:
- 用 OpenCode 或 Codex 都可以
- 内容质量一样,只是慢一点

**如果未来要 URL 安装**:
- 等 `npx skills add` 平台原生支持
- 当前用 git clone 或 zip 包

## 8. 一句话总结

> **降级不影响"内容质量",但影响"产出效率"和"架构能力"。**
>
> **内容质量由"主 Agent + 方法论 + 质量门控"保证,不依赖子 Agent。**
>
> **产出效率由"并行能力"决定,降级时变慢但内容完整。**

**openPRD V5.0 透明承诺:三个平台产出内容 100% 一致,只是速度不同。**
