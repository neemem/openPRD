# 20 · G0 需求清晰度门控(Requirements Clarity Gate)

> 📖 **使用说明**:G0 是 openPRD **Step 0 与 Step 1 之间**的"防错门控"。主 Agent 在用户答完 Step 0 的 4 问后,**必须**跑 G0 ——把答案复述一遍、检测冲突、用户确认后才进 Step 1。本文档是 G0 的**唯一规则来源**。
>
> **Why**:实际项目 30% 失败源于"我以为你要 X 但你要 Y"。G0 把答案复述一遍,让用户在 Step 1 大规模启动前发现偏差,避免下游 subagent 基于错误前提工作。

---

## 0. 文档目的

- **复述机制**:把 Step 0 的自动判断 + 用户回答合并成"最终画像卡片"
- **冲突检测**:5 类常见互斥 + 5 类常见强冲突
- **透明度**:让用户看到"主 Agent 替你做了什么判断",有错即纠
- **G0 与其他门控的关系**:G0 是 Step 0 的"形式门控",G1-G3 是 Step 1-3 的"形式+产物门控"

---

## 1. G0 触发条件

| 条件 | 是否必跑 |
|------|----------|
| 完整模式(后端+页面级+BI 全选) | ✅ **必跑** |
| 简化模式(纯前端) | ✅ 必跑(更轻量) |
| 单角色项目(只选 1 类受众) | ✅ 必跑 |
| 用户消息 < 30 字,主 Agent 自动判断 < 3 个维度 | ⚠️ 推荐跑(主 Agent 必跑) |
| 35 题调研启动前 | ✅ **必跑**(防止 35 题基于错误前提) |

---

## 2. G0 执行步骤(主 Agent 必走)

### Step 1:合并"自动判断 + 用户回答"为最终画像

```markdown
📋 最终需求画像(G0 复述)
─────────────────────────────
[自动判断]          [用户回答]              [合并后]
需求主题:   -      兰花品种行情(用户原话)  →  兰花品种行情 ✅
目标用户:   80%   C 端兰花爱好者            →  C 端兰花爱好者 ✅
项目类型:   100%  全新(用户原话含"全新")   →  全新 ✅
交付模式:   70%   用户选"完整+页面级+无BI" →  完整·页面级·无 BI ✅
产出范围:   60%   用户选"产品+技术"        →  2 类受众(运营/合规降级) ✅
行业合规:   100%  一般(无金融/医疗关键词)  →  一般 ✅
时间预算:   90%   "下周要 demo"(用户原话)  →  1-2 周 MVP ⚠️ 建议改 1-3 天 demo
─────────────────────────────
```

**为什么"画像卡片"形式**:用户在 1 张表上看到"我答了什么 + Agent 判了什么 + 最终是什么",视觉上 5 秒内能确认。

### Step 2:互斥检测(5 类)

| # | 互斥项 | 检测方法 | 处理 |
|---|--------|----------|------|
| 1 | 页面级 + 简化模式 + 仅前端 | 3 选都成立 | 提示"页面级方案建议完整模式,是否切换?" |
| 2 | 含 BI + 简化模式 + 功能级 | 3 选都成立 | 提示"BI 看板建议页面级 + 完整模式" |
| 3 | 仅技术受众 + 含产品文档 | 互斥 | 提示"选'技术'后 06-PRD 会降级,PM 看不到完整版" |
| 4 | 选合规排除 + 项目含金融/医疗 | 强冲突 | 拒绝继续,提示"金融/医疗项目不能排除合规" |
| 5 | 海外/跨境 + 选合规排除 | 强冲突 | 拒绝继续,提示"海外项目必须含 GDPR/数据本地化" |

### Step 3:用户确认

主 Agent 把"最终画像 + 互斥检测结果"展示给用户,问 1 个简单问:
> "**G0 复述完毕**:上述画像是否准确?有 3 种回复方式——
> 1. 回"OK" / "继续" → 立即进 Step 1 启动 35 题 + 5 路研究
> 2. 指出某项不对 → 主 Agent 修正画像后再复述
> 3. 想补充背景 → 直接说,主 Agent 更新画像"

---

## 3. G0 失败处理

| 失败情况 | 处理 |
|----------|------|
| 用户拒绝 G0,坚持原答案 | 主 Agent 记录"用户已确认 G0 冲突仍坚持",继续推进(尊重用户意愿) |
| 用户未回复 G0(等 30 分钟) | E7 策略:礼貌重试 1 次,仍无回 → 用合理默认值 + 标"待确认" |
| G0 检测出强冲突(金融/医疗 排除合规) | 拒绝继续,解释风险,等用户修正 |
| G0 检测出软冲突(简化+页面级) | 不拒绝,只提示,让用户决策 |
| 主 Agent 自动判断有误 | G0 透明度机制兜底,用户在 G0 复述阶段修正 |

---

## 4. G0 与 G1-G3 的关系

```
Step 0 (4 问)  →  G0 (复述+冲突)  →  Step 1 (35 题+5 路)
   ↑ 形式门控         ↑ 形式门控          ↑ 形式门控 G1
                                            ↑ 产物门控 G1.5
                                            ↑ 阶段门控 G2
                                            ↑ 全局门控 G3
```

**G0 是"开门前最后一次对齐"**。G1-G3 是"已经开工后的质量检查"。

---

## 5. G0 输出格式(主 Agent 必出)

```markdown
## G0 需求清晰度门控 · 复述报告

📋 最终画像:
- 需求主题: {主题}
- 目标用户: {用户}
- 项目类型: {全新/二期}
- 交付模式: {完整/简化}·{页面级/功能级/仅后端}·{含BI/无BI}
- 产出范围: {N} 类受众({产品/技术/运营/合规})
- 行业合规: {一般/行业监管/跨境}
- 时间预算: {1-3天demo/1-2周MVP/1-3月完整/3月+长期}

🤖 主 Agent 自动判断覆盖: {N} 个维度 / 7(主题/用户/项目类型/模式/方案/BI/行业)

⚠️ 互斥检测:
- {通过 / 警告: 软冲突 N 项 / 阻止: 强冲突 N 项}

✅ 如无误请回"OK"开始 Step 1;有修正请直接指出。
```

---

## 6. 常见误用(Gotchas)

| Gotcha | 症状 | 修正 |
|--------|------|------|
| G-01: 跳过 G0 直接进 Step 1 | 用户答完 4 问后主 Agent 立即启动 subagent | 必跑 G0,即使只有 1 个用户回答 |
| G-02: G0 复述漏写"自动判断覆盖度" | 用户不知道 Agent 做了什么 | 必出"主 Agent 自动判断覆盖 N/7 维度" |
| G-03: G0 强冲突还强行继续 | 金融项目排除合规,主 Agent 仍跑 | 强冲突必须拒绝,等用户修正 |
| G-04: G0 复述用技术语言 | "schema_version: 1.1" 用户看不懂 | 用画像卡片+emoji,业务语言 |
| G-05: G0 触发后用户回"无所谓" | 模糊回答 | 必重述画像,直到用户明确"OK" |
