# Mock 数据真实性确认（详细参考）

## 概述

Mock 数据必须贴合用户真实业务场景，不能凭空捏造。

---

## 1. Mock 数据调研（前置步骤）

在产出任何文档前，必须与用户确认以下 Mock 数据要素：

| # | 问题 | 目的 |
|---|------|------|
| 1 | **数据类型有哪些？** | 确定 Mock 的数据实体（如：用户、订单、设备） |
| 2 | **数据规模多大？** | 确定 Mock 数据量级（10条/100条/10000条） |
| 3 | **数据分布特征？** | 确定正常/异常/边界数据比例 |
| 4 | **业务规则是什么？** | 确定状态流转、计算公式等业务逻辑 |
| 5 | **敏感数据处理？** | 确定是否脱敏、加密存储等要求 |

---

## 2. Mock 数据不确认时的处理

如果用户无法确认 Mock 数据要素，按以下策略处理：

| 场景 | 策略 |
|------|------|
| 数据类型不确定 | 使用通用实体（id, name, status, createTime, updateTime） |
| 数据规模不确定 | 按最小可用量级 Mock（列表 ≤ 20条） |
| 业务规则不确定 | 标注"待确认"，使用通用状态流转 |
| 敏感数据不确定 | 默认按脱敏处理，不暴露真实信息 |

---

## 3. Mock 数据产出标准

```
✅ 正确示例：
- 用户列表：[{ id: 1, name: "张三", phone: "138****1234", status: 1 }]
- 状态枚举：{ 1: "启用", 2: "禁用" }

❌ 错误示例：
- 使用真实手机号：13812345678
- 使用可识别姓名：张三丰
- 使用真实身份证：110101199001011234
```
