# 13 · 自检清单单一来源（Quality Self-Check）

> 📖 **使用说明**：本文档是 openPRD 全流程**自检规则的唯一来源**。
> 所有"自检 / 验收 / 完整性检查 / G1/G2/G3 门控 / 最终交付前自检"均引用本文档。
> 各模板的「段完成度自检」「必含项自检」「页面级自检清单」**不再独立维护**，均按本文档 §1 规范生成。
>
> **Why**：原体系下"自检清单"在 18+ 处重复（每份模板 + 主 SKILL.md + 05-quality-gates.md + 06a-06h 各段 + 14-行业/竞品/标杆/合规/技术 5 份），更新遗漏即出现"漏检 / 重检"。

---

## 0. 文档目的

- **统一三档自检粒度**：模板级 / 阶段级 / 全局级
- **每项检查有"通过标准 + 验证方法"**：避免"通过/不通过"主观判断
- **单一引用点**：主 SKILL.md、05-quality-gates.md、00-INDEX、各模板均按本文档 §1 规范生成
- **可执行**：`grep` / `cat` / `node scripts/check-consistency.js` 即可机械化验证

## 1. 模板级自检（每份文档产出后立即执行）

> 适用范围：每份 33 份模板在 subagent 完成后立即自检。
> 模板内统一章节名：`## 📋 段完成度自检`（06a-06h 用此格式）/ `## 段完成度自检（更新）`（扩展模板）/ `## 自检清单`（14-调研类）。

### 1.1 通用必含项（所有模板）

| # | 检查项 | 通过标准 | 验证方法 |
|---|--------|----------|----------|
| 1 | 段落完整 | 无 `TBD / TODO / 待补充 / 根据需求定` 占位符 | `grep -c "TBD\|TODO\|待补充" <doc>` ≤ 3 |
| 2 | 数据可追溯 | ≥ 80% 数据有来源（数据来源 / 调研 / 用户回答） | 抽样 10 个数字，≥ 8 个有引用 |
| 3 | Mermaid 图 | 涉及流程 / 状态 / 时序 / ER 时必含 | 视觉化需求模板：≥ 1 张 Mermaid |
| 4 | 加粗关键 | 关键发现 / 验收标准 / 必填字段 / 错误码已加粗 | 扫读 30 秒能定位 P0 范围 |
| 5 | 代码可执行 | 代码块可复制运行（占位符 ≤ 1 个） | 抽查 3 个代码块，无 `<待填写>` |
| 6 | 自检章节 | 文档末尾含"段完成度自检"或"必含项自检" | `grep "## 段完成度自检\|## 必含项自检\|## 自检清单" <doc>` = 1 |
| 7 | 文档定位 | 头部含 `> 📖 文档定位` `> 🎯 核心问题` `> 📚 配套文档` | `grep -E "📖\|🎯\|📚" <doc>` ≥ 3 |
| 8 | 模式标注 | 头部含 `> 所属阶段` `> 价值判定` | 视觉确认 |

### 1.2 文档族专项必含项

| 文档族 | 必含项 | 自检章节位置 |
|--------|--------|----------------|
| **06a-06h**（PRD 拆段） | 每段完成度自检 + 段价值说明 | 文档末尾 `## 📋 段完成度自检` |
| **07a-07c**（测试拆段） | 段契约 + 用例数量 + 场景比例 | 文档开头 `## 段契约` + 末尾用例统计 |
| **14-行业分析** | 9 大章节 + 波特五力 + PEST + 行业术语表 | `## 10. 自检清单` |
| **14-竞品分析** | 11 章节 + 波特五力 + SWOT + Battle Card | `## 11. 自检清单` |
| **14-标杆研究** | 6 步法 + ≥ 15 来源 + 60%+ 12 个月内 | `## 10. 自检清单` |
| **14-合规清单** | 8 章节 + ≥ 30 项 Checklist + 引用条款 | `## 12. 自检清单` |
| **14-技术趋势** | 7 大领域 + 本项目选型建议 | `## 10. 自检清单` |
| **用户调研报告** | 35 题 / 9 组 / 必答 18 题 | `## 16. 自检清单` |
| **02 / 04 / 05 / 10 / 13 / FigmaMake** | 章节完整 + 关键决策标注 ⭐ | `## 段完成度自检` / `## 段完成度自检（更新）` |

### 1.3 模板级"自检章节"统一格式

```markdown
## 📋 段完成度自检   (06a-06h 用此)

- [ ] X.1 子章节：必含项 N1
- [ ] X.2 子章节：必含项 N2
- [ ] ...
- [ ] X.n 子章节：必含项 Nn

**段价值**：本段产出后，下游段（XX-YY）可基于 YYY 展开 ZZZ。
- 06x：基于 YYY 展开 ZZZ
- ...

---

## 段完成度自检（更新）  (扩展模板 04/10/13/FigmaMake/06h 用此)

- [x] 1-N. 原基础章节完整
- [x] N+1. 新增章节 1
- [x] N+2. 新增章节 2
- ...
```

> **Note**：所有模板的段完成度自检**保留各模板的具体检查项**（这是模板特有的），但**章节格式与位置**按本规范统一。

## 2. 阶段级自检（G1 / G2）

> 适用范围：每个阶段（调研 / 设计 / 开发 / 测试 / 文档产出）完成后，主 Agent 立即执行。
> 详细门控逻辑见 `reference/05-quality-gates.md` §2-§3；本节只列"阶段通过条件"。

### 2.1 G1 - 单 subagent 阶段门控

> 时机：每个 subagent（行业分析 / 竞品分析 / 文档产出等）完成后立即验证。

| subagent | 通过条件 | 验证方法 |
|----------|----------|----------|
| `openPRD-user-research` | 35 题回答 ≥ 80% + Q33 复述确认 + Q35 下一步确认 | `grep "Q[0-9]" <report>` ≥ 35 |
| `openPRD-industry-analysis` | 9 大章节 / 波特五力 / PEST / 行业术语表 | `grep "波特五力\|PEST"` ≥ 2 |
| `openPRD-competitor-analysis` | 11 章节 / 波特五力 / SWOT / Battle Card | `grep "SWOT\|Battle Card"` ≥ 2 |
| `openPRD-deep-research` | 6 步法 / ≥ 15 来源 / 60%+ 12 个月内 | 数引用条目 |
| `openPRD-regulation-compliance` | 8 章节 / ≥ 30 项 Checklist / 引用条款 | `grep -c "^- \[" checklist` ≥ 30 |
| `openPRD-tech-trends` | 7 大领域 / 本项目选型建议 | `grep "^## " report` ≥ 7 |
| `openPRD-context7-integration` | 库 ID 来源 / 文档版本 / 代码示例 | 引用块末尾含"来源标注" |
| `openPRD-minimax-integration` | Prompt 模板 / 失败降级 / 成本警告 | 调用块末尾含"成本警告" |
| `openPRD-chrome-devtools-integration` | 资产路径 / 性能审计 / 资源释放 | 调用块末尾含"资源释放" |
| `openPRD-document-generation` | 6+1 文档完整 / 跨文档字段一致 | 各文档末尾含"自检清单" |

**G1 失败处理**：

| 失败类型 | 处理 |
|----------|------|
| 占位符 > 20% | 立即修复，标记为"草稿" |
| 数据无来源 | 立即补来源或删除 |
| Mermaid 图错误 | 立即修复（语法问题） |
| 跨文档字段不一致 | 主 Agent 协调修复（详见 `reference/15-five-field-crosscheck.md`） |

### 2.2 G2 - 阶段产出门控

> 时机：每个阶段（调研 / 设计 / 开发 / 测试）完成后。

#### 调研阶段

| 项 | 通过标准 |
|----|----------|
| 用户调研 | 35 题回答 ≥ 80% + Q33 复述确认 + Q35 下一步确认 |
| 行业研究 5 路 | 5 份独立报告 + 1 份综合洞察 + 冲突解决 |
| 综合置信度 | ≥ 70% 关键结论有数据/引用支撑 |
| 输出物 | 用户调研 + 14-行业 + 14-竞品 + 14-标杆 + 14-合规 + 14-技术趋势 |

#### 设计阶段

| 项 | 通过标准 |
|----|----------|
| 项目整体说明 | 5W 齐全 + 商业模式画布 + 价值主张 |
| PRD 06 主控 | 16 章索引 + 段间契约清晰 |
| PRD 06a-06h | 各段完成度自检 ≥ 90% |
| 06-PRD 文档量 | ≥ 50KB（页面级） / ≥ 15KB（功能级） |
| 跨文档字段一致 | 用户调研 / 行业分析 / PRD 中关键字段名一致 |

#### 开发阶段

| 项 | 通过标准 |
|----|----------|
| 接口文档 | ≥ 80% 核心 API 有完整定义 |
| 数据库设计 | 实体字段与接口字段一致 |
| 架构设计 | 7 大领域（前端/后端/数据/AI/安全/通信/国产化）齐全 |
| Mock 数据 | 字段类型与接口一致 |
| 任务拆分 | 每个任务 1-3 天 + 验收标准 |

#### 测试阶段

| 项 | 通过标准 |
|----|----------|
| 测试用例 | P0 覆盖 100% + 边界异常 + 性能 |
| 验收标准 | 来自 PRD AC + Given-When-Then |

### 2.3 阶段级门控失败处理

1. 主 Agent **不得**进入下一阶段
2. 列出该阶段所有不通过项
3. 重新调用对应 subagent
4. 重新跑 G2 门控
5. 全部通过后才进入下一阶段

## 3. 全局级自检（G3 + 最终交付报告）

> 适用范围：所有阶段完成后，主 Agent 在向用户汇报前必须完成。
> 完整门控详见 `reference/05-quality-gates.md` §4；本节列出"全局通过条件"。

### 3.1 完整性

| 检查项 | 通过标准 | 验证方法 |
|--------|----------|----------|
| 文档数量 | 完整模式 14 份 / 简化模式 9 份（详见 `reference/14-mode-tag-table.md`） | 对照清单逐项打勾 |
| 模板应用 | 06-PRD 拆 8 段（页面级）/ 单文档（功能级） | 检查 06a-06h 存在性 |
| 子技能产出 | 所有调用的 subagent 有产物 | 检查 `*-docs/` 目录 |
| 跨文档引用 | 上下游有引用链接 | 抽查 10 个引用 |

### 3.2 价值密度

| 检查项 | 通过标准 |
|--------|----------|
| 每份文档有"回答的核心问题" | 17/17 明确 1 句话 |
| 占比空壳 | < 5%（TBD/待补充/根据需求定）|
| 跨文档字段一致 | 5 个关键字段（userId/status/必填规则/错误码/P0 功能名）— 详见 `reference/15-five-field-crosscheck.md` |
| 文档量级 | 06-PRD ≥ 50KB（页面级）/ 14-行业分析 ≥ 30KB / 02-项目整体 ≥ 20KB |
| 可执行性 | 用户拿到 02/06/05/06d 后能立即评审/排期 |
| 价值链清晰 | 调研→设计→开发→测试→管理 上下游明确 |
| 跨文档研究严谨性 | 14-行业分析 ≥ 15 来源且 60%+ 12 个月内 / 用户调研 ≥ 2 种方法 + ≥ 5 个用户引用 |
| 外部能力使用 | 涉及具体技术栈时调用 context7 / 涉及视觉化时调用 minimax / 涉及实时数据时调用 chrome-devtools |

### 3.3 5 个关键字段交叉对比（必查）

> 详见 `reference/15-five-field-crosscheck.md`（单一来源）

| 字段 | 出现位置 | 验证方法 |
|------|----------|----------|
| 字段名（如 `userId`） | 06-PRD / 03-接口 / 12-DB / 11-Mock | 4 处拼写完全一致 |
| 枚举值（如 `status`） | 06-PRD / 03-接口 / 11-Mock / 07-测试 | 4 处枚举值完全一致 |
| 必填规则 | 06-PRD / 03-接口 | 必填标记一致 |
| 错误码 | 03-接口 / 09-后端 / 07-测试 | 错误码与测试用例对应 |
| P0 功能名 | 06-PRD / 07-测试 / 05-任务 | P0 功能名在三处一致 |

### 3.4 页面级专项门控（仅页面级方案触发时）

| 项 | 通过标准 |
|----|----------|
| 页面清单 | 所有页面 11 字段齐全（名称/路径/类型/是否需登录/页面状态/SEO/分享/相关US/关联UC/预加载/权限）|
| 组件库 | 每个基础组件 5 维度（变体/尺寸/状态/交互/可访问性）|
| 交互矩阵 | 每个页面 8 状态（初始/加载/空数据/错误/网络错误/服务器错误/无权限/成功）|
| 数据模型 | ERD + 实体字段 + 索引 |
| 接口契约 | 请求/响应/错误码/超时/重试/幂等 |
| 页面流转图 | Mermaid，含正常/异常分支 |
| 业务规则 | 每条规则编号 + 触发 + 逻辑 + 输出 + 异常 |
| 状态机 | 核心对象（订单/客户/支付）含状态枚举 + 转移 |
| 埋点 | 每个 P0 功能含事件设计 |
| 测试用例 | 每个 P0 US 含 Given-When-Then |

### 3.5 最终自检报告（向用户汇报前的固定输出）

> 主 Agent 必须输出以下报告，未输出则视为"未自检"。

```markdown
## 自检报告

### G1 单 subagent 门控
- [✅/❌] 用户调研：35 题 / 9 组 / 必答 18 题
- [✅/❌] 行业分析：9 章节 / 波特五力 / PEST
- [✅/❌] 竞品分析：11 章节 / Battle Card
- [✅/❌] 标杆研究：6 步法 / 15+ 来源
- [✅/❌] 合规清单：8 章节 / 30+ Checklist
- [✅/❌] 技术趋势：7 领域 / 本项目选型
- [✅/❌] 文档产出：6+1 完整 / 跨文档一致
- [✅/❌] PRD 拆段：06a-06h 完整 / 50KB+

### G2 阶段门控
- [✅/❌] 调研阶段：用户调研 + 5 路行业研究
- [✅/❌] 设计阶段：项目整体 + 16 章 PRD
- [✅/❌] 开发阶段：接口 + DB + 架构 + 任务
- [✅/❌] 测试阶段：P0 100% + 边界异常

### G3 全局门控
- [✅/❌] 完整性：N 份文档 / 期望 N
- [✅/❌] 价值密度：5 项交叉对比
- [✅/❌] 可执行性：02/06/05 立即可评审
- [✅/❌] 外部能力：context7 / minimax / chrome-devtools

### 5 个关键字段交叉对比  (详见 reference/15-five-field-crosscheck.md)
- [✅/❌] `userId` 在 4 处一致
- [✅/❌] `status` 在 4 处一致
- [✅/❌] 必填规则一致
- [✅/❌] 错误码一致
- [✅/❌] P0 功能名一致

### 结论
✅ **自检通过，可以交付** / ⚠️ **需修复 N 项后交付**
```

### 3.6 自检不通过时（修复流程）

| 优先级 | 类型 | 修复策略 |
|--------|------|----------|
| P0 | 文档缺失 / 关键字段不一致 | 立即补全 |
| P1 | 数据无来源 / 自检清单缺失 | 补全 + 标注 |
| P2 | 视觉化不足 / 价值密度偏低 | 增加图表 / 引用 |
| P3 | 格式细节 | 批量优化 |

**主 Agent 不得在自检不通过时向用户汇报**。修复后重新自检，全部通过才汇报。

## 4. 自检 vs 反模式（不要混淆）

| 维度 | 自检（本文档） | 反模式（`reference/07-anti-patterns.md`） |
|------|----------------|------------------------------------------|
| 范围 | openPRD 流程产出物 | 通用 PRD / 工程实践 |
| 性质 | 流程级"必含项" | 反例库（症状→修复） |
| 触发 | 每次文档产出后 | 文档评审/Code Review 时 |
| 关系 | 反模式命中 ≥ 1 次 → 自检不通过 | 自检是反模式的"上一步" |

> 跨技能常见错误（pitfalls）见 `reference/16-common-pitfalls.md`（不与本表重复）。
