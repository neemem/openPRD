# 07 · openPRD 反模式清单（Anti-Patterns）

> 把 openPRD 实施过程中"踩过的坑"系统化。本文档是**质量门的前置关卡**——任何一项反模式命中 ≥ 1 次，必须修复后才能进开发。

---

## 0. 文档目的

openPRD 体系下，PRD / API / DB / 前端 / 测试 5 份核心文档是开发执行的"唯一真相源"。但实际项目中，**80% 的返工来自文档本身的反模式**——而不是开发能力。

**反模式 = 反复出现的错误模式 + 可识别的特征 + 可量化的修复方法**。

本文档目的：

- 汇总 20+ 常见反模式
- 每个反模式给"特征 → 后果 → 修复"三段式
- 提供"自检清单"作为发布前最后一道关

> 反模式不是"个人风格问题"，而是"组织能力问题"——必须用规范、模板、CI 三件套系统化解决。

---

## 1. 占位符过多（TBD / 待补充 / TODO / 根据需求定）

**特征**：

- 文档中 `TBD` / `TODO` / `待补充` / `根据需求定` 出现 ≥ 5 处
- 占总字段比例 > 20%
- 关键路径（如 P0 接口、状态机、错误码）也存在占位符

**后果**：

- 开发"猜"接口字段，10 处猜 5 处错
- 测试无依据，用例无法编写
- 上线后才发现"返回值未定义"

**修复**：

- 占位符必须替换为真实值
- 实在无法确定，**标记为 `Blocked` 并指定 owner + 截止日期**
- 用 `<!-- TODO(owner, 2026-06-10) -->` 注释 + 任务系统追踪

**自检**：`grep -c "TBD\|TODO\|待补充" prd/*.md` 输出 ≤ 3。

---

## 2. 字段不一致（userId vs user_id vs uid）

**特征**：

- 同一字段在多份文档命名不同
- userId / user_id / uid / userID 混用
- status / state / Status 混用

**后果**：

- 前端按 A 命名取值，后端返回 B，联调报错
- 反序列化失败、空值、默认值异常

**修复**：

- 建立 `fields-master.json`（详见 `06-cross-document-consistency.md` §7.1）
- 跑 `check-consistency.js` 脚本（见同文档 §4）
- 全替换 + 二次 Review

**自检**：脚本 0 报错。

---

## 3. 空壳文档（仅目录无内容）

**特征**：

- 文档有 5 级标题，但每节只写一行概述
- 表格只有表头，无数据行
- 代码块只有注释，无实际逻辑

**后果**：

- 看起来"完整"，实则"零信息"
- 开发无法执行
- 客户/上级检查时被发现，信任崩塌

**修复**：

- 每节至少 50 字 + 1 个表格/代码块/示例
- 模板用 `<示例>` 标记必填项（不是可选）
- 评审时拒绝"空壳文档"过审

**自检**：随机抽 3 节，字数 < 30 字 = 空壳。

---

## 4. 流程图缺失（业务复杂却无图）

**特征**：

- 涉及多角色 / 多状态 / 多分支的业务（如订单、审批、支付）
- 文档中无任何 Mermaid / PlantUML 图
- 文字描述 > 5 段仍说不清流程

**后果**：

- 角色理解不一致
- 状态机缺失导致代码 if-else 嵌套 5 层
- 测试用例漏关键分支

**修复**：

- 强制规则：涉及 ≥ 2 角色 / ≥ 3 状态 / ≥ 1 决策点 → 必须配流程图
- 用 Mermaid `flowchart` / `sequenceDiagram` / `stateDiagram-v2`
- 详见 `reference/08-mermaid-syntax.md`

**自检**：复杂业务文档 Mermaid 代码块 ≥ 2 个。

---

## 5. 加粗规范（关键内容未加粗）

**特征**：

- 关键决策（P0 范围、必填字段、错误码、风险项）无视觉强调
- 全文无加粗 / 大量加粗（每个字都加粗）

**后果**：

- 阅读者抓不住重点
- 评审时反复问"这个是 P0 吗？"

**修复**：

- **加粗三要素**：决策、约束、风险
- P0 功能名、必填字段、错误码、关键时间点必须加粗
- 避免"全文加粗"——选择性强 + 视觉层次

**自检**：扫读 PRD，能在 30 秒内找出 P0 范围。

---

## 6. Mock 数据失真（与真实业务脱节）

**特征**：

- 文档示例：`userId: "123"`
- 真实业务：UUID v4，36 字符
- 文档示例：金额 100，真实业务含小数（100.00）

**后果**：

- 开发按 123 实现，集成时类型不匹配
- 测试按 100 断言，金额比较失败

**修复**：

- Mock 数据必须**与真实业务 1:1**（长度、类型、格式、边界值）
- 边界值必须覆盖：空、极小、极大、特殊字符、Unicode
- 用 `01-mock-data.md`（见 reference）规范

**自检**：抽样 5 个 mock，与真实接口 schema 对比一致。

---

## 7. 跨文档拼写差异（中英文、缩写、单复数）

**特征**：

- 同一概念：用户 / 用户(User) / usr / USER 混用
- 单复数：order / orders 混用
- 大小写：getUser / getuser / GetUser 混用

**后果**：

- 文档可读性差
- 国际化（i18n）key 不一致
- 命名风格不统一

**修复**：

- 维护"术语表"（中英文对照）`glossary.md`
- 单复数规则：DB 表复数（users）、API 单数（/user/123）
- 大小写：URL kebab-case，TS camelCase，Java camelCase，DB snake_case

**自检**：术语表覆盖核心概念 ≥ 90%。

---

## 8. 枚举值不一致（status 字段 4 处不一致）

**特征**：

- 状态值在 PRD / API / DB / 前端 / 测试中写法不同
- 例：PRD "已激活"，DB `enabled`，前端 `Active`，测试 `ENABLED`

**后果**：

- 前端 i18n 翻译错位
- 后端 enum 转换异常
- 测试断言失败

**修复**：

- 枚举值统一**全大写 + 下划线**（PENDING / ACTIVE）
- 建立 `enums-master.json`
- 5 处文档必须从 master 引用

**自检**：5 处文档枚举值 grep 完全一致。

---

## 9. 错误码与文档不对应

**特征**：

- API 文档列出错误码 A001，但 PRD §6 异常处理未引用
- 后端抛出 B001，文档无定义
- 错误码总表位置不统一（散落各 PRD）

**后果**：

- 客户端无法处理未定义错误
- 用户体验：英文栈错误直接展示

**修复**：

- 独立 `error-codes.md`（错误码总表）
- 6 位数字 + 模块位
- 任何新增错误码必须先更总表，再写代码

**自检**：所有 API 文档错误码 = 总表子集。

---

## 10. 测试用例缺失 P0 覆盖

**特征**：

- PRD 列出 6 个 P0 功能
- 测试用例 P0 覆盖 < 80%
- P0 关键路径（登录、支付）无正向用例

**后果**：

- 上线后 P0 功能故障
- 客户首批反馈即问题

**修复**：

- 测试用例模板强制 §P0 覆盖矩阵
- 每个 P0 功能 ≥ 5 个用例（正常 + 异常 + 边界 + 权限 + 性能）
- 评审时拒绝"P0 覆盖不足"过审

**自检**：P0 覆盖矩阵 100%。

---

## 11. 任务拆分粒度过粗

**特征**：

- 任务写"实现用户管理"（5 天）
- 不可拆分、不可分配
- 进度无法追踪

**后果**：

- 进度黑盒
- 风险集中爆发
- Code Review 一次 1000 行

**修复**：

- 任务 ≤ 2 人天
- 必须可独立验收
- 含：输入 / 输出 / 工时 / 依赖 / Owner

**自检**：每个任务可独立 PR。

---

## 12. ADR 缺失

**特征**：

- 关键决策（数据库选型、状态机方案、缓存策略）无 ADR
- 团队成员质疑"为什么用 Redis 而不是 Memcached"
- 无历史决策记录

**后果**：

- 重复讨论同一问题
- 新人无法理解设计意图
- 决策可追溯性差

**修复**：

- 关键决策 ≥ 10 个 → 写 ADR
- 模板：背景 / 决策 / 后果 / 备选方案
- 存于 `docs/adr/0001-xxx.md`

**自检**：核心模块均有 ADR 引用。

---

## 13. 性能基准无量化

**特征**：

- PRD 写"性能好"、"响应快"
- 无具体数字（≤ 500ms / 1000 TPS）

**后果**：

- 测试无依据
- 上线后性能不达标
- 客户投诉"慢"

**修复**：

- 性能指标必须量化：P95 / P99 / QPS / 并发数
- 测试环境跑 benchmark
- 性能基线存档

**自检**：所有"性能"段含具体数字。

---

## 14. 验收标准模糊（"性能好"而非"P95 ≤ 500ms"）

**特征**：

- 验收写"界面美观"、"功能完整"
- 无可测量的标准

**后果**：

- 客户与开发对"完成"理解不同
- 验收扯皮

**修复**：

- 验收标准用 SMART 原则（Specific / Measurable / Achievable / Relevant / Time-bound）
- 例：登录 P95 ≤ 500ms、错误率 < 0.1%、支持 1000 并发

**自检**：每条验收可被测试用例验证。

---

## 15. 时间估算虚高 / 虚低

**特征**：

- 工时拍脑袋（"3 天"）
- 实际 1 周或 1 天
- 缺乏依据

**后果**：

- 项目延期或过度承诺
- 团队失去信任

**修复**：

- 用 Planning Poker 估算
- 区分乐观 / 期望 / 悲观值
- 缓冲 20%

**自检**：估算值与实际偏差 < 30%。

---

## 16. 状态机与代码不对应

**特征**：

- PRD 状态机 4 个状态
- 代码 if-else 实际 6 个状态（多了"已退款"）
- 转移条件缺失

**后果**：

- 代码即真相，文档作废
- 后续维护者困惑

**修复**：

- 状态机 Mermaid 图 + 代码 enum 严格 1:1
- Code Review 校验状态机实现
- 任何新增状态先更文档

**自检**：grep 代码 enum 与 PRD 状态 1:1。

---

## 17. 权限矩阵缺失

**特征**：

- 多角色（管理员 / 普通用户 / 访客）
- 文档无角色 × 操作 × 资源矩阵

**后果**：

- 越权访问安全漏洞
- 权限测试遗漏

**修复**：

- 维护 `rbac-matrix.md`
- 行=角色，列=资源/操作，单元格=允许/拒绝/条件
- 测试用例覆盖矩阵

**自检**：所有角色均有显式权限定义。

---

## 18. 国际化（i18n）未规划

**特征**：

- 文档中所有文案硬编码中文
- 无 i18n key 设计
- 后续多语言支持需重写

**后果**：

- 国际化重做成本高
- 文案散落各组件难维护

**修复**：

- 所有用户可见文案提取为 i18n key
- 用 `zh-CN.json` / `en-US.json` 等文件管理
- 文档中标注文案 key

**自检**：PRD 中所有用户可见文案均含 i18n key。

---

## 19. 监控/告警/日志缺失

**特征**：

- PRD 描述业务功能
- 无监控指标、告警规则、日志规范

**后果**：

- 上线后无观测
- 故障定位慢

**修复**：

- 关键接口监控（QPS / P95 / 错误率）
- 告警规则（错误率 > 1% → 通知）
- 日志规范（结构化 JSON、traceId、userId）

**自检**：核心接口均有监控埋点。

---

## 20. 依赖与版本未锁定

**特征**：

- 文档推荐"使用 Redis"
- 无版本号、配置示例
- 实际部署用 Redis 5，文档基于 Redis 7

**后果**：

- 环境不一致
- "我这边正常"问题频发

**修复**：

- 关键依赖锁定版本（Redis 7.2.4 / Node 20.11.0）
- `package.json` / `requirements.txt` 提交
- 文档标注版本

**自检**：所有外部依赖均含版本号。

---

## 21. 兼容性/降级方案缺失

**特征**：

- 新接口上线，老接口未保留
- 移动端旧版本无兼容方案
- 服务降级未设计

**后果**：

- 老用户崩溃
- 故障扩散

**修复**：

- API 版本管理（/v1/、/v2/）
- 灰度发布策略
- 降级方案（缓存降级、限流降级）

**自检**：破坏性变更均有版本管理 + 降级方案。

---

## 22. 文档与代码版本脱节

**特征**：

- 文档 v1.0，代码已 v2.3
- 文档半年未更新

**后果**：

- 文档作废，新人迷惑

**修复**：

- 文档与代码同仓库
- CI 校验：PR 必含文档更新（如有行为变更）
- 季度文档审计

**自检**：文档更新日期 ≤ 30 天。

---

## 自检清单

发布前最后检查（每项必填）：

- [ ] **占位符**：`grep -c "TBD\|TODO\|待补充"` ≤ 3
- [ ] **字段一致**：check-consistency 脚本 0 错
- [ ] **空壳**：抽样 3 节字数 ≥ 50
- [ ] **流程图**：复杂业务 Mermaid ≥ 2 个
- [ ] **加粗**：30 秒能定位 P0 范围
- [ ] **Mock 数据**：与真实业务 1:1
- [ ] **拼写**：术语表覆盖率 ≥ 90%
- [ ] **枚举值**：5 处文档 grep 一致
- [ ] **错误码**：API 错误码 ⊆ 总表
- [ ] **P0 覆盖**：测试矩阵 100%
- [ ] **任务粒度**：每个任务 ≤ 2 人天
- [ ] **ADR**：核心决策均有 ADR
- [ ] **性能量化**：所有"性能"段含数字
- [ ] **验收 SMART**：可被测试验证
- [ ] **时间估算**：有依据 + 缓冲
- [ ] **状态机**：代码 enum = 文档状态
- [ ] **权限矩阵**：所有角色显式定义
- [ ] **i18n**：用户可见文案有 key
- [ ] **监控告警**：核心接口有埋点
- [ ] **依赖版本**：所有依赖有版本号
- [ ] **兼容性**：破坏性变更均有方案
- [ ] **文档新鲜度**：≤ 30 天

> **22 项全绿** = 可以发布。**红项 ≥ 1** = 必须修复。
